# OpenCode大型项目代码理解插件测试文档
## 文档说明
本文档用于指导LLM生成测试代码并执行自动化测试，覆盖插件所有功能模块、接口和场景，保障插件符合需求和设计规范。

> **版本说明（重要）**
>
> - 当前代码仓库已演进到 **V2.0 独立 CLI + V2.1 LLM 原生解析 + V2.2 解析结构优化 + V2.3 路径查询 Skill 与黑名单优化** 形态。
> - **第 12 章为人工测试暴露问题分析与测试缺口补救（必读）**：说明进度条、文件/目录结果完整性等为何未被自动化发现，并给出 P0/P1 补救用例的详细设计与实施要求。
> - **第 10 章为 V2.3 适配后的最新测试规范，第 9 章为 V2.2，第 6~7 章为 V2.0/V2.1，应作为编写和维护测试代码的主参考。**
> - **第 3~5 章主要保留 V1.0/OpenCode 插件形态下的测试设计，仅作历史参考，不再作为当前实现的强制约束。**
> - 如 V1.0 测试用例与 V2.x 设计/实现冲突，以 V2.x 最新章节说明和当前实现为准。
> - V2.3 引入的关键变更：集中式索引文件（`analysis-index.json`）、`resolve` 子命令替代 `query`、Agent Skill 自动生成与部署、三层黑名单过滤规则重构。**涉及 `query` 子命令的历史测试用例均已废弃，以 `resolve` 替代。**

## 1. 测试方法
### 1.1 测试分层策略
| 测试层级 | 测试范围 | 测试方式 | 覆盖率要求 |
|---------|---------|---------|-----------|
| 单元测试 | 单个类、接口、函数的逻辑正确性 | 隔离依赖（Mock外部服务），白盒测试 | 核心领域层100%，服务层90%+ |
| 集成测试 | 模块间调用、外部依赖交互 | 真实依赖（Git/存储等），灰盒测试 | 核心流程100%覆盖 |
| 系统测试 | 完整功能流程、用户场景 | 黑盒测试，模拟用户操作 | 所有需求场景100%覆盖 |
| 性能测试 | 解析速度、资源占用、并发能力 | 压力测试、负载测试 | 满足需求文档性能指标 |
| 可靠性测试 | 异常处理、断点恢复、数据一致性 | 故障注入测试 | 异常场景100%覆盖 |
| 兼容性测试 | 跨平台、多语言、OpenCode版本适配 | 多环境矩阵测试 | 支持的所有环境覆盖 |

### 1.2 测试执行流程
1. 单元测试优先：核心逻辑修改后先执行单元测试
2. 集成测试验证：模块集成后执行集成测试验证交互逻辑
3. 系统测试验证：完整功能验证，覆盖所有用户场景
4. 非功能测试验证：性能、可靠性、兼容性测试
5. 回归测试：每次迭代后全量执行已有测试用例

## 2. 测试环境
### 2.1 硬件环境
| 环境类型 | 配置要求 |
|---------|---------|
| 最低配置 | 2核CPU，4GB内存，50GB磁盘空间 |
| 推荐配置 | 8核CPU，16GB内存，100GB SSD磁盘 |
| 性能测试配置 | 16核CPU，32GB内存，200GB SSD磁盘 |

### 2.2 软件环境
| 软件 | 版本要求 |
|-----|---------|
| 操作系统 | Windows 10+/macOS 12+/Ubuntu 20.04+ |
| Node.js | >=16.0.0 |
| OpenCode | >=1.0.0 |
| Git | >=2.30.0 |
| 测试框架 | Jest >=29.0.0 |

### 2.3 测试项目准备
| 测试项目类型 | 说明 | 用途 |
|-------------|------|------|
| 小型项目 | 10个以内代码文件，包含TS/JS/Python等多种语言，非Git项目 | 基础功能测试 |
| 中型项目 | 100-500个代码文件，Git仓库，包含多分支、多提交记录 | 增量解析、集成测试 |
| 大型项目 | 1000+个代码文件，复杂目录结构，多模块依赖 | 性能测试、并发测试 |
| 异常项目 | 包含语法错误文件、敏感文件、超大文件（10MB+） | 异常场景测试 |
| 空项目 | 空目录、仅包含配置文件 | 边界场景测试 |

### 9.9 多次运行与一致性测试

#### 用例ID：ST-V22-RUN-001  
**测试场景**：全量 → 增量 → 增量 多轮运行一致性  

- **前置条件**：  
  - 准备一个 Git 项目，包含多级目录与若干代码文件。

- **测试步骤**：  
  1. 第一次：在干净状态执行 `code-analyze analyze . --mode full`。  
  2. 修改少量文件并提交，执行第二次 `--mode incremental`。  
  3. 再修改另一批文件并提交，执行第三次 `--mode incremental`。  
  4. 在第三次之后，单独在当前 commit 上再执行一次全量解析，对比结果。

- **预期结果**：  
  1. 每一轮增量解析仅对真实变更文件及受影响目录重新解析。  
  2. 多轮运行中对象级进度统计始终一致、合理。  
  3. 第三轮结束后的结果与在同一 commit 上直接全量解析的结果一致（允许时间戳等元信息不同，但结构和内容需一致）。

## 11. E2E 端到端测试实施规范（V2.3+）

本章节补充**端到端（E2E）测试的统一约束与落地方式**，回答如下问题：

- 如何通过**全局环境变量**控制「是否 Mock LLM」以及真实 LLM 的 API Key 读取方式；
- 如何构造并复用一个**100+ 文件的大型测试项目**，系统性验证黑名单过滤与索引完整性；
- E2E 测试中对 **Skill 部署与验证** 的统一策略；
- 所有 E2E 测试的**目录结构、命名约定**与**临时目录清理机制**。

### 11.1 LLM Mock / 真实调用全局控制规范

#### 11.1.1 控制维度与环境变量约定

E2E 测试必须通过**单一全局环境变量**来控制是否 Mock LLM，禁止在测试代码中通过硬编码或本地开关「偷偷」决定是否访问真实 LLM。

- 统一环境变量名：`CODE_ANALYZE_E2E_LLM_MODE`
- 允许取值：
  - `"mock"`：使用 Mock LLM 客户端（**默认值**，CI 中强制使用）；
  - `"real"`：使用真实 LLM 服务。
- 未设置或设置为非法值时的行为：
  - 视为 `"mock"`，即默认使用 Mock LLM，保证 CI 与本地的安全性与稳定性。

#### 11.1.2 测试代码中的读取逻辑

所有 E2E 测试在初始化 LLM 相关依赖时，必须遵循以下伪代码逻辑：

```ts
const llmMode = (process.env.CODE_ANALYZE_E2E_LLM_MODE || 'mock').toLowerCase();
const useMockLLM = llmMode !== 'real';
```

- 当 `useMockLLM === true` 时：
  - 所有 CLI 调用或内部服务初始化都通过配置/环境变量将 LLM 调用「指向」本地 Mock 客户端；
  - E2E 仅验证「流程正确」「索引与文件系统一致」等，不验证真实模型输出内容。
- 当 `useMockLLM === false` 时：
  - 允许 E2E 调用真实 LLM，但**必须动态读取 API Key**，且决不写死在代码中。

#### 11.1.3 真实 LLM API Key 的安全读取规范

当 `CODE_ANALYZE_E2E_LLM_MODE=real` 时，E2E 测试必须：

1. **从固定的私有文件中动态读取 Key**，禁止硬编码：
   - 相对路径：`tests/priv_data/api_key.txt`
   - 绝对路径示例（以当前仓库根为 `c:\Software\Project\skills-any-code`）：  
     `c:\Software\Project\skills-any-code\tests\priv_data\api_key.txt`
2. 读取规则：
   - 文件可能包含多行信息，标准格式要求：
     - 第 1 行：`provider=<可选，模型提供方标识>`（可选，如果未来支持多 provider）
     - 第 2 行：`api_key=<真实 key>` 或直接为 API Key 本身；
   - 为简化当前实现，可以约定：
     - **只读取首个非空、非注释行**（忽略以 `#` 开头的注释），将其作为 API Key 字符串；
   - 伪代码示例：

   ```ts
   import * as fs from 'fs-extra';
   import * as path from 'path';

   async function loadE2EApiKey(projectRoot: string): Promise<string | null> {
     const keyFile = path.join(projectRoot, 'tests', 'priv_data', 'api_key.txt');
     if (!(await fs.pathExists(keyFile))) return null;

     const content = await fs.readFile(keyFile, 'utf-8');
     const lines = content
       .split(/\r?\n/)
       .map(l => l.trim())
       .filter(l => l && !l.startsWith('#'));
     return lines[0] || null;
   }
   ```

3. **严禁行为**：
   - 在测试代码中写死任何形式的 Key、Token、密钥片段；
   - 将 `tests/priv_data/api_key.txt` 内容提交到版本库（应在 `.gitignore` 中忽略该文件或整个 `tests/priv_data` 目录）。

#### 11.1.4 CLI 调用与 LLM 配置的约定

在 E2E 测试中需要调用 CLI 时：

- Mock 模式：
  - 通过环境变量或配置文件将 LLM Base URL 指向本地假服务（如 `http://127.0.0.1:0/mock-llm`），但实现层可以直接绕过 HTTP，只在内部拦截；
  - 或使用 `--llm-base-url`/`--llm-api-key dummy` 等参数，但真实调用层必须被 Mock 截断。
- Real 模式：
  - 先从 `tests/priv_data/api_key.txt` 读取 Key；
  - 再通过环境变量 `CODE_ANALYZE_LLM_API_KEY` 或 CLI 参数 `--llm-api-key` 传入；
  - 任何测试代码中都不能出现明文 Key 字符串。

> **建议**：CI 默认不设置 `CODE_ANALYZE_E2E_LLM_MODE` 或显式设置为 `"mock"`，仅在开发者本地调试或专门的 Nightly 任务中手动启用 `"real"`，并在对应环境安全地配置 `tests/priv_data/api_key.txt`。

#### 11.2.1 目录结构

所有端到端测试统一放在 `tests/e2e/` 目录下，与 `unit/integration/system` 等层次并列。例如：

```ts

  // ...
});
```

这样可以保证测试代码与测试文档之间的**一一可追踪性**。

### 11.3 E2E 临时目录与数据清理策略

#### 11.3.1 临时目录根路径约定

所有 E2E 运行时产生的临时代码项目、结果文件、日志等必须集中存放在仓库内的一个固定目录下，避免污染工作区其它位置：

- 统一根目录：`tests/e2e/.tmp/`
  - Windows 绝对路径示例：  
    `c:\Software\Project\skills-any-code\tests\e2e\.tmp\`

#### 11.3.2 每个用例的子目录规范

- 每个 E2E 用例在 `.tmp` 下创建专属子目录，命名规则建议为：
  - `<用例ID>-<简短英文场景名>`，例如：
    - `ST-V23-FLOW-001-full-index-skill/`
    - `ST-V23-FLOW-003-blacklist-consistency/`
    - `ST-V23-FLOW-008-large-project-index/`
- 完整路径示例：

```text
tests/e2e/.tmp/
  ST-V23-FLOW-001-full-index-skill/
  ST-V23-FLOW-003-blacklist-consistency/
  ST-V23-FLOW-008-large-project-index/
```

#### 11.3.3 清理策略

1. **测试前统一清理**：
   - 在 E2E 全局 `beforeAll` 或 Jest 全局 setup 中执行：

   ```ts
   await fs.remove(path.join(projectRoot, 'tests', 'e2e', '.tmp'));
   await fs.ensureDir(path.join(projectRoot, 'tests', 'e2e', '.tmp'));
   ```

   - 保证每次 E2E 执行前 `.tmp` 目录是**干净的**，不会被上一次测试的残留干扰。

2. **测试过程中的行为约束**：
   - 所有 E2E 用例必须只在各自的 `.tmp/<case>` 子目录下创建 Git 仓库、复制/生成测试项目、运行 CLI；
   - 禁止在仓库根目录或其它路径直接修改用户真实工程文件。

3. **测试后的可选保留策略**：
   - 默认策略：**不在每个用例后单独清理子目录**，由统一的「测试前清理」保证一致性；
   - 若需要在失败时保留临时目录以便排查，可增加一个辅助环境变量：
     - `CODE_ANALYZE_E2E_KEEP_TMP_ON_FAIL=true`；
     - 测试工具在捕获到用例失败时仅输出临时目录路径提示，不再做额外清理。

### 11.4 100+ 文件大型测试项目设计（含黑名单配置）

#### 11.4.1 目录与文件布局建议

在某个 E2E 用例下（例如 `ST-V23-FLOW-008-large-project-index`），在对应的 `.tmp` 子目录中生成如下结构：

```text
<tmpRoot>/
  large-project/
    .git/                # 初始化为 Git 仓库
    .gitignore
    .code-analyze-ignore # 可选，用于测试项目级黑名单
    package.json
    README.md
    docs/
      intro.md
      design.md
    config/
      default.yaml
      local.yaml
    scripts/
      build.sh
      deploy.sh
    src/
      index.ts
      app.ts
      modules/
        module01/
          index.ts
          service01.ts
          service02.ts
        module02/
          ...
        ...
        module10/
          index.ts
          service01.ts
          service02.ts
      shared/
        utils/
          util01.ts
          ...
        types/
          type01.ts
          ...

      client.generated.ts

      .placeholder

```

- 通过 `modules`/`shared` 等子目录，可以轻松生成 **100+ 代码文件**（例如 10 个 module，每个 module 3–5 个 `.ts` 文件）。
- `docs/`、`config/`、`generated/`、`node_modules/`、`logs/` 等用于测试不同层级和语义的黑名单规则。

#### 11.4.2 黑名单相关文件与规则设计

1. **全局黑名单（配置 `analyze.blacklist`）**：
   - 默认值中已包含：
     - `*.md`、`*.txt`、`*.yaml`/`*.yml`、`docs/`、`node_modules/`、`.git/`、`.code-analyze-result/` 等；
   - 在该大项目中：
     - `README.md`、`docs/**`、`config/*.yaml`、`logs/*.log` 等都应被全局黑名单或 `.gitignore`/`.code-analyze-ignore` 覆盖。

2. **项目级黑名单（`.code-analyze-ignore`）** 示例：

```text
# 忽略所有 generated 代码与日志
generated/
logs/

# 但显式允许 src 下的 generated 类型定义
!src/shared/types/generated-types.ts
```

3. **`.gitignore` 示例**：

```text
node_modules/
dist/
coverage/
*.log
```

该组合可覆盖以下场景：
- 来自全局配置的模式（如 `*.md`、`docs/`）；
- 来自项目级 `.code-analyze-ignore` 的模式（如 `generated/`、`logs/` + 否定模式）；
- 来自 `.gitignore` 的模式（如 `node_modules/`、`*.log`）。

#### 11.4.3 大型项目 E2E 用例中的断言要点

在对应的 E2E 测试（例如 `v23-large-project-index.e2e.test.ts`）中，需要重点断言：

1. **解析覆盖率**：
   - `analysis-index.json` 中的条目个数 ≈ 「代码类文本文件数 + 目录数」，明显小于项目总文件数（因为配置/文档/日志等被过滤）。
2. **黑名单有效性**：
   - 任意抽查几个命中黑名单的路径（如 `README.md`、`docs/intro.md`、`config/default.yaml`、`logs/app.log`、`generated/client.generated.ts`），在索引中不存在，`resolve` 返回 `N/A`；
   - 未被黑名单过滤的 `src/**/*.ts` 均在索引中，`resolve` 返回的路径存在。
3. **索引完整性**：
   - 抽样检查 `src/modules/**`、`src/shared/**` 中若干文件，索引与实际 Markdown 一一对应；
   - 对某些中间目录（如 `src/modules/module01/`）的 `index.md` 存在，并在索引中有 `type="directory"` 条目。

### 11.5 E2E 中对 Skill 的使用策略

#### 11.5.1 默认策略：开启 Skill 生成

结合需求文档中对 Agent Skill 的定位以及你对测试覆盖的要求，**E2E 默认应开启 Skill 生成与校验**：

- 除非特定用例明确声明 `--no-skills`，所有 E2E 的 `code-analyze analyze` 调用都**不加 `--no-skills`**，从而触发：
  - `.agents/skills/code-query/`（opencode/cursor/codex）；
  - `.claude/skills/code-query/`（claude）  
  的自动生成与更新。

#### 11.5.2 Skill 校验粒度

为平衡执行时间与覆盖度，建议：

- 对以下**核心 Flow 用例**必须做 Skill 校验：
  - `ST-V23-FLOW-001`（全量 → 索引 → Skill → resolve）；
  - `ST-V23-FLOW-002`（增量 → 索引与 Skill 更新一致性）；
  - `ST-V23-FLOW-007`（自定义 output-dir 下的 Skill 路径与索引一致性）。
- 对一些仅关注性能或黑名单的 E2E 用例（如 `ST-V23-FLOW-008` 大型项目索引完整性）：
  - 可以选择保留 Skill 生成但**只做轻量断言**（例如仅检查 Skill 目录存在），或在用例描述中明确 `--no-skills` 以缩减执行时间。

#### 11.5.3 Skill 与 CLI resolve 结果对比

在每个包含 Skill 校验的 E2E 用例中，推荐至少对 1–3 个路径做如下对比：

1. 使用 CLI：

```bash
code-analyze resolve <absolute-source-path> --project <projectRoot> --output-dir <outputDir?>
```

2. 使用 Skill 目录下脚本：

```bash
node .agents/skills/code-query/scripts/resolve.js <absolute-source-path>
```

3. 断言两个调用的 stdout 完全一致（均为同一个 Markdown 绝对路径或同为 `N/A`）。

## 12. 人工测试暴露问题分析与测试缺口补救（必读）

本章基于人工测试发现的严重问题（进度条不更新、仅生成目录 md 无文件 md、目录内容空洞），对「为何自动化测试未暴露」进行根因归类，并给出详细的补救用例设计与实施要求。**所有新增用例均需在后续迭代中实现并纳入回归，不得仅保留在文档中。**

### 12.1 问题与根因摘要

#### 12.1.1 问题一：进度条不更新、文件数与扫描状态错误

| 维度 | 说明 |
|------|------|
| **现象** | 解析过程中进度条始终显示「已处理: 0/100 文件」，百分比 0%，无变化；解析结束后仍显示 0 个文件。 |
| **实现根因** | 1）CLI 层（`src/cli.ts`）仅调用 `progressBar.start(100, 0, …)`，总对象数写死为 100，且从未向 `AnalysisAppService.runAnalysis()` 传入进度回调；<br>2）应用层 `fullAnalysis()` 参数中无 `onProgress`/`progressCallback`；<br>3）领域层遍历与解析过程中未设计「每处理完一个对象即上报进度」的接口，因此进度条与真实处理状态从未打通。 |
| **需求/设计依据** | 需求文档 10.2（进度展示规范）、测试文档 UT-TERM-001、ST-V22-PROG-001/002 均要求进度条实时反映「已完成对象数/总对象数」、百分比与剩余时间。 |
| **测试未暴露原因** | **测试实现缺口**：文档中已定义 UT-TERM-001、ST-V22-PROG-001/002，但代码库中无任何测试引用 `progressBar`、`UT-TERM-001` 或 `ST-V22-PROG`，即进度相关用例从未落地为自动化测试。 |

#### 12.1.2 问题二：仅生成目录的 md、无文件级 md，且目录内容空洞

| 维度 | 说明 |
|------|------|
| **现象** | 解析结果中仅有各层目录的 `index.md`，没有源文件对应的 `{文件名}.md`；目录 `index.md` 内容为「包含 0 个文件，N 个子目录」等空洞描述，无实际文件列表与功能描述。 |
| **可能根因** | 1）该目录下确实无代码文件（仅子目录）时，`fileResults.filter(Boolean).length` 为 0，目录摘要正确但无文件 md；<br>2）若整棵结果树均无文件 md，则可能是：所有文件的 LLM 解析在真实环境下失败（超时/限流/格式错误），仅目录聚合在写入；或某种路径/仓库结构下存储或遍历存在边界 bug。 |
| **需求/设计依据** | 需求文档 4.2 存储规范、UT-STO-001/002、ST-FULL-001 等要求「每个代码文件生成对应分析文件」「目录生成 index.md 且内容包含子项与功能描述」。 |
| **测试未暴露原因** | **测试覆盖不足**：UT-CLI-011 仅断言根目录 `index.md` 与「共分析 ≥5 个文件」，未断言每个源文件均有对应 `.md`；UT-CLI-012 仅覆盖根目录下无后缀文件的 .md，未覆盖「多级目录下每个代码文件都有对应 .md」及「目录 index.md 内容有意义」。集成测试使用 Mock LLM，所有文件解析成功，无法暴露真实 LLM 全失败或路径结构边界问题。 |

| 问题 | 文档环节 | 测试脚本实现环节 | 实现环节 |

## 13. Markdown 结果结构一致性测试（新增）

本章针对最近由人工测试暴露、但未被自动化用例覆盖的四类 Markdown 结果问题，补充统一的结构一致性测试规范，确保后续版本在 CI 阶段即可发现类似问题。

### 13.1 测试范围与目标

- **文件级 Markdown 结构**：
  - 约束文件结果中一级标题（章节）集合，禁止出现未定义的新章节（如 `[依赖]`）。
  - 保证 `[类定义]` 等关键章节在**同一文件内只出现一次**，所有类定义都聚合在该章节下。
- **目录级 Markdown 结构与内容质量**：
  - 目录 `index.md` 必须包含非空的功能描述，不能仅有“包含 N 个子目录/文件”的机械统计语句。
  - 目录下的子目录/文件数量与 `index.md` 中的描述应大致一致。
- **文件 / 目录命名冲突处理**：
  - 明确当存在 `index.xxx` 源文件与同名目录时，结果 Markdown 的命名规则和映射关系。
  - 保证不会出现目录结果覆盖文件结果、或反之的情况，索引文件 `analysis-index.json` 与磁盘文件保持一一对应。

### 13.2 文件级 Markdown 结构测试

#### 13.2.1 用例 ID：UT-FILE-MD-STRUCT-001（文件章节白名单）

- **测试场景**：文件级 Markdown 的一级标题只允许出现在预定义白名单中。
- **前置条件**：
  - 准备一个包含 1–2 个代码文件的小型测试项目，执行全量解析（使用 Mock LLM）。
- **测试步骤**：
  1. 读取某个文件的结果 Markdown（例如 `src/foo.ts → .code-analyze-result/src/foo.md`）。
  2. 提取所有一级标题（例如匹配 `^#\s*\[(.+?)\]`）。
  3. 将标题文本与预定义白名单集合进行比较。
- **预期结果**：
  - 一级标题集合**仅包含**如下章节（示例，具体以实现约定为准）：
    - `[基本信息]`
    - `[概述]`
    - `[功能描述]`
    - `[类定义]`
    - `[全局变量]`
    - `[全局函数]`
  - **不允许**出现白名单之外的章节（如 `[依赖]` 等）。
- **断言要点**：
  - `actualHeadings ⊆ allowedHeadings`；
  - `unexpectedHeadings.length === 0` 时用例通过，否则失败。

### 14. `.code-analyze-result` 被继续解析且未被测试发现

#### 14.1 现象与期望

- **现象**：在 Windows 环境下，解析项目时，输出目录 `.code-analyze-result` 已配置在全局黑名单中（`analyze.blacklist`），但仍然被递归遍历并尝试解析，最终在结果中出现对 `.code-analyze-result` 目录下文件的二次解析。
- **期望**：
  - `.code-analyze-result/` 目录及其所有子目录、子文件应被严格排除在解析范围之外：
    - 不参与对象计数；
    - 不调用 LLM；
    - 不生成对应 Markdown；
    - 不写入索引 `analysis-index.json`。

#### 14.2 根因分析

- 黑名单本身的配置与 `BlacklistService.load` 逻辑正确，问题出在「调用方传入的路径风格」：
  - 在 `AnalysisService.countObjects` 与 `fullAnalysis` 的遍历逻辑中，使用：
    - `const relativePath = path.relative(projectRoot, fullPath)`；
  - 在 Windows 环境下，`relativePath` 会含有 `\` 分隔符，例如：
    - `.code-analyze-result\\index.md`；
  - 但黑名单模式使用的是 `.gitignore` 风格（`/` 分隔符），并依赖 `ignore` 库进行匹配。
- 单元测试中对 `isIgnored` 的所有调用都使用了 POSIX 风格路径：
  - 如 `'node_modules/foo'`、`'dist/bundle.js'`、`'src/index.ts'` 等；
  - CI 环境多为 Linux/macOS，`path.relative` 返回路径也为 `/`，因此该问题在自动化测试中无法复现。

#### 14.3 补充自动化测试用例设计

**1）新增 Windows 路径风格单元测试**

- **位置**：`tests/unit/filter/blacklist-filter.test.ts`
- **新增用例 ID**：`UT-V23-BL-WINPATH-001: Windows 分隔符路径仍能被黑名单过滤`
- **步骤**：
  1. 构造 `BlacklistService` 实例；
  2. 调用 `load(['.code-analyze-result/'], someProjectRoot)`；
  3. 调用：
     - `service.isIgnored('.code-analyze-result\\index.md')`；
     - `service.isIgnored('src\\index.ts')`；
  4. 断言：
     - 前者为 `true`；
     - 后者为 `false`。
- **目的**：防止后续有人重构时再次忽略对 Windows 分隔符的归一化。

**2）新增集成/系统测试：输出目录不会被再次解析**

- **位置建议**：`tests/system/full-analysis.test.ts` 或新建 `tests/system/blacklist-output-dir.test.ts`
- **用例 ID 建议**：
  - `ST-V23-BL-OUTDIR-001: 输出目录 .code-analyze-result 不参与解析`
- **场景**：
  1. 在一个空目录下创建简单项目（1~2 个 `.ts` 文件）；
  2. 执行一次分析，生成 `.code-analyze-result`；
  3. 再次执行一次分析；
  4. 遍历实际参与解析的源路径集合（可通过：

  5. 断言：

- **平台要求**：
  - 本系统用例要求在 Windows 环境下至少执行一次（本地或 CI 矩阵），以验证路径分隔符兼容性。

### 15.4 并发「当前对象列表」自动化测试（对应需求文档 12.4）

#### 15.4.1 帮助函数：解析「当前对象」快照

```ts
interface CurrentObjectsSnapshot {
  groups: string[][]; // 每次出现「当前对象」块时的一组路径列表
}

function extractCurrentObjects(output: string): CurrentObjectsSnapshot {
  const lines = stripAnsi(output).split(/\r?\n/);
  const groups: string[][] = [];
  let current: string[] | null = null;

  for (const raw of lines) {
    const line = raw.trim();
    if (/^当前对象[:：]?$/.test(line)) {
      if (current && current.length > 0) groups.push(current);
      current = [];
      continue;
    }
    if (current && line && (line.includes('/') || line.includes('\\'))) {
      current.push(line);
    }
  }
  if (current && current.length > 0) groups.push(current);

  return { groups };
}
```

### 15.5 Token 使用统计自动化测试（对应需求文档 12.5）

#### 15.5.1 帮助函数：解析 Token 行

```ts
interface TokenSnapshot {
  prompt: number;
  completion: number;
  total: number;
}

function extractTokenSnapshots(output: string): TokenSnapshot[] {
  const lines = stripAnsi(output).split(/\r?\n/);
  const snapshots: TokenSnapshot[] = [];
```

```ts
interface LLMConfig {
  baseUrl: string;
  apiKey: string;
  model: string;
  timeoutMs: number;
}

class LLMClient {
  constructor(private cfg: LLMConfig) {}

  private validateConfig() {
    if (!this.cfg.baseUrl || !this.cfg.apiKey || !this.cfg.model) {
      throw new LLMConfigError('LLM_CONFIG_INCOMPLETE', {
        missing: {
          baseUrl: !this.cfg.baseUrl,
          apiKey: !this.cfg.apiKey,
          model: !this.cfg.model,
        },
      });
    }
  }

  async connectTest(): Promise<void> {
    this.validateConfig();

    try {
      const res = await httpPost(
        `${this.cfg.baseUrl}/chat/completions`,
        {
          model: this.cfg.model,
          messages: [{ role: 'system', content: 'health-check' }],
          max_tokens: 1,
        },
        { timeout: this.cfg.timeoutMs },
      );

      if (res.status === 401) {
        throw new LLMConnectError('LLM_AUTH_FAILED');
      }
      if (res.status === 404) {
        throw new LLMConnectError('LLM_MODEL_NOT_FOUND');
      }
      if (res.status < 200 || res.status >= 300) {
        throw new LLMConnectError('LLM_UNEXPECTED_STATUS', { status: res.status });
      }
    } catch (e) {
      if (e instanceof TimeoutError || e instanceof NetworkError) {
        throw new LLMConnectError('LLM_NETWORK_ERROR', { cause: e });
      }
      throw e;
    }
  }
}
```

### 16.5 缓存容量上限与近似 LRU 回收测试设计

```ts
interface FileHashCacheOptions {
  cacheDir: string;
  maxSizeMb: number; // 0 表示禁用
}

class FileHashCache {
  constructor(private opts: FileHashCacheOptions) {}

  private async getDirSize(): Promise<number> {
    const files = await fs.promises.readdir(this.opts.cacheDir);
    let total = 0;
    for (const f of files) {
      const stat = await fs.promises.stat(path.join(this.opts.cacheDir, f));
      if (stat.isFile()) total += stat.size;
    }
    return total;
  }

  private async enforceLimit() {
    if (this.opts.maxSizeMb <= 0) {
      return;
    }

    let total = await this.getDirSize();
    const maxBytes = this.opts.maxSizeMb * 1024 * 1024;
    if (total <= maxBytes) return;

    const files = await fs.promises.readdir(this.opts.cacheDir);
    const fileStats: { path: string; mtimeMs: number; size: number }[] = [];

    for (const f of files) {
      const full = path.join(this.opts.cacheDir, f);
      const stat = await fs.promises.stat(full);
      if (!stat.isFile()) continue;
      fileStats.push({ path: full, mtimeMs: stat.mtimeMs, size: stat.size });
    }

    fileStats.sort((a, b) => a.mtimeMs - b.mtimeMs);

    for (const f of fileStats) {
      if (total <= maxBytes) break;
      try {
        await fs.promises.unlink(f.path);
        total -= f.size;
      } catch (e) {
        logger.warn(`删除缓存文件失败: ${f.path}`, e);
      }
    }
  }

  async set(hash: string, value: any): Promise<void> {
    if (this.opts.maxSizeMb === 0) {
      return;
    }

    await fs.promises.mkdir(this.opts.cacheDir, { recursive: true });
    await this.enforceLimit();

    const filePath = path.join(this.opts.cacheDir, `${hash}.json`);
    await fs.promises.writeFile(filePath, JSON.stringify(value), 'utf8');
  }

  async get(hash: string): Promise<any | null> {
    if (this.opts.maxSizeMb === 0) {
      return null;
    }
    const filePath = path.join(this.opts.cacheDir, `${hash}.json`);
    if (!fs.existsSync(filePath)) return null;
    const raw = await fs.promises.readFile(filePath, 'utf8');
    return JSON.parse(raw);
  }
}
```

   - 统一强调：自 V2.5 起，默认黑名单中包含各类图片资源扩展名，对「统计对象数」「过滤列表」相关测试的预期值需排除这些文件。
