## 文档目的

本文件记录对 `docs/测试文档.md`（以 **V2.5** 为最终版本基线）与当前仓库 `tests/` 自动化测试代码的**逐条对照评审结果**，输出：

- **缺失用例清单**：文档要求存在，但测试代码未实现/未以对应用例 ID 落地
- **不一致清单**：测试已存在，但断言/退出码/行为与文档预期不一致
- **多余/过期清单**：测试代码存在，但不属于 V2.5 最终全集（或与 V2.5 全局前提冲突）

> 说明：本评审按你的要求 **只输出差异/缺失**，不在此文件中直接修改测试代码。

## 不一致清单（用例已存在，但与文档预期不一致）

### 1) `UT-V23-RESOLVE-004`（第 10 章）

- **文档预期**：索引文件不存在时，stderr 有错误信息，**退出码为 1**
- **当前集成测试实现**：`tests/integration/cli/resolve.test.ts`
  - 断言为“索引不存在时输出 `N/A`”，并允许当前实现“exit 0 + N/A”
- **结论**：测试断言与文档预期冲突，无法保证符合最终版本规范。

### 2) 第 16 章 16.1 全局前提在多处 CLI 测试中未被满足

文档要求：

- V2.5 起除 `init` 外，`analyze/resolve/config` 在配置文件不存在时必须立即失败（不再隐式创建）
- 正常解析成功类用例必须提供有效 `llm.base_url/api_key/model`，并先通过 connectTest

现状：

- 多个系统/集成测试直接运行 `analyze`（甚至 `npm run build` 后立即执行），**没有**统一 `init` + `-c <tmpConfig>` 的前置步骤，存在“依赖本机已有配置文件”的隐含条件

## 多余/过期/冲突用例清单（不属于 V2.5 最终全集，或与最终前提冲突）

- `tests/unit/config.test.ts` 中的 `UT-CFG-001~004`（以及同文件中部分 `UT-V23-CFG-*` 仍调用 `load()` 并假设可成功）

> 结果：同仓库内存在“load 必须失败不落盘”和“load 会落盘生成”的两套互斥测试目标，影响对最终版本行为的信心与可维护性。
