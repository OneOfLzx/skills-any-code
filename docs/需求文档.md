# OpenCode大型项目代码理解插件需求文档

## 1. 文档概述

### 1.1 目的

本文档定义OpenCode大型项目代码理解插件的功能需求、交互流程、存储规范和开发约束，确保插件符合OpenCode生态规范，实现大型项目的自动化代码解析、结构化分析和知识沉淀能力。

### 1.2 范围

覆盖插件全部功能模块：命令入口、代码解析Agent、查询Skill、增量解析、结果存储，不包含插件的打包发布流程和运营相关需求。

### 1.3 目标用户

- 大型项目开发者：快速理解陌生项目结构
- 架构师：自动化生成项目架构文档
- 代码审计人员：批量梳理代码逻辑和依赖关系

### 1.4 术语定义


| 术语         | 定义                              |
| ---------- | ------------------------------- |
| 全量解析       | 首次解析项目时对所有代码文件进行完整分析的过程         |
| 增量解析       | 后续解析时仅分析自上次解析以来发生变更的文件/目录的过程    |
| 分析结果元数据    | 记录解析版本、git commit信息、解析时间等元数据的文件 |
| 共同祖先commit | 两个分支分叉前的最后一个共同提交记录，用于计算跨分支的代码差异 |


## 2. 产品总览

插件以OpenCode命令形式提供入口，通过专用Agent完成项目代码的自动化解析，生成结构化的分析结果并本地存储，同时提供查询Skill供LLM调用获取项目知识。支持全量解析和增量解析两种模式，自动适配git项目的变更，采用目录级并行解析策略保障大型项目的解析性能。

## 3. 功能详细需求

### 3.1 命令入口实现

#### 3.1.1 命令注册

- 命令名称：`/analyze-project`
- 命令参数：

  | 参数名         | 类型                         | 必填  | 说明                  |
  | ----------- | -------------------------- | --- | ------------------- |
  | path        | string                     | 否   | 指定解析的项目根路径，默认当前工作目录 |
  | mode        | enum: `full`/`incremental` | 否   | 解析模式，默认自动检测         |
  | depth       | number                     | 否   | 解析深度，默认无限制          |
  | concurrency | number                     | 否   | 并行解析并发数，默认CPU核心数*2  |

- 实现方式：遵循OpenCode插件工具注册规范，使用`tool()`方法注册，参数采用Zod schema验证

#### 3.1.2 命令触发交互

- 用户在对话框输入`/analyze-project`即可触发解析
- 触发后自动检测当前目录是否为git项目，输出提示：`[项目代码理解] 开始解析项目：{项目名称}，当前模式：{全量/增量}，并发数：{n}`
- 解析过程中实时输出进度：`已解析文件：{已解析数}/{总文件数}，已完成目录：{已完成目录数}，正在并行解析目录：[dir1, dir2, ...]`
- 解析完成后输出结果入口：`解析完成，整体项目文档已生成：{存储路径}/PROJECT_SUMMARY.md`

### 3.2 代码解析Agent实现

Agent采用两轮解析架构：第一轮自底向上遍历分析，第二轮自顶向下校验修正。

#### 3.2.1 第一轮：自底向上遍历分析

- 遍历规则：
  - 从指定根路径开始递归遍历项目目录树
  - 自动跳过：支持读取项目`.gitignore`规则，并自动跳过解析
  - 仅分析文本代码文件：支持所有主流编程语言，由LLM自动识别语言类型并解析，无特定文件后缀限制
  - 并行策略：无依赖关系的同级目录采用并行解析，同一目录下的文件串行解析（避免IO冲突），不同根分支的目录完全并行处理
- 文件级分析逻辑（叶子节点）：
  1. 提取文件基本信息：文件名、语言类型、代码行数、依赖引入
  2. 分析文件核心作用：100字以内的功能描述
  3. 提取所有类定义：类名、继承关系、核心方法、属性列表
  4. 生成类图代码：采用Mermaid类图语法
  5. 生成核心流程时序图：提取文件内核心方法调用链，生成Mermaid时序图代码
- 目录级分析逻辑：
  1. 等待当前目录下所有并行解析的子目录/文件任务完成
  2. 汇总当前目录下所有文件和子目录的分析结果
  3. 总结目录核心作用：说明该目录在项目中的定位
  4. 梳理目录结构：列出所有子目录和核心文件的功能映射

#### 3.2.2 第二轮：自顶向下校验修正

- 修正规则：
  1. 从根目录开始，基于父目录的分析结果审阅子目录/文件的分析结论
  2. 检查不一致：子目录/文件描述与父目录定位冲突的情况
  3. 检查遗漏：核心功能模块未被正确识别的情况
  4. 检查错误：类继承关系、方法调用链描述错误的情况
- 修正逻辑：
  1. 对存在问题的分析结果进行修正，保留修正前后的对比记录
  2. 修正后同步更新对应文件/目录的分析结果
  3. 所有修正记录写入`ANALYSIS_MODIFICATION_LOG.md`文件

#### 3.2.3 最终整体分析

- 生成项目整体结构图：Mermaid架构图，展示项目分层、模块依赖关系
- 生成项目核心流程图：Mermaid流程图，展示项目核心业务流程/数据流转
- 生成项目总结：包含项目定位、核心功能、技术栈、架构特点、代码规模等信息

#### 3.2.4 分析结果内容规范

- 类图/时序图/架构图统一采用Mermaid语法，确保可直接渲染
- 所有描述采用结构化Markdown格式，使用统一的标题层级
- 代码片段使用对应语言的语法高亮

### 3.3 查询Skill实现

#### 3.3.1 Skill定义

- Skill名称：`project-code-query`
- Skill描述：获取指定路径的文件/目录的代码分析结果
- 实现方式：遵循OpenCode Skill规范，在`.opencode/skills/project-code-query/SKILL.md`中定义

#### 3.3.2 Skill参数


| 参数名  | 类型                               | 必填  | 说明                 |
| ---- | -------------------------------- | --- | ------------------ |
| path | string                           | 是   | 需要查询的文件/目录相对项目根的路径 |
| type | enum: `summary`/`full`/`diagram` | 否   | 查询类型，默认`summary`   |


#### 3.3.3 Skill返回结果

- `summary`模式：返回文件/目录的核心功能描述
- `full`模式：返回完整的分析结果，包括所有类定义、方法、类图、时序图
- `diagram`模式：仅返回相关的图代码（类图/时序图/架构图）

### 3.4 增量解析实现

#### 3.4.1 解析状态记录

- 每次解析完成后，保存最近50次解析的git状态：commit hash、分支名称、解析时间戳
- 状态记录文件路径：`{存储根目录}/.analysis_metadata.json`
- 元数据结构：
  ```json
  {
    "project_root": "/path/to/project",
    "last_analyzed_at": "2026-03-09T12:00:00.000Z",
    "git_commits": [
      {"hash": "a1b2c3d4e5f6", "branch": "main", "analyzed_at": "2026-03-09T12:00:00.000Z"},
      {"hash": "b2c3d4e5f6a1", "branch": "feature/login", "analyzed_at": "2026-03-08T10:00:00.000Z"},
      // ... 最多保存最近50条解析记录
    ],
    "analysis_version": "1.0",
    "analyzed_files_count": 128
  }
  ```

#### 3.4.2 增量解析逻辑

- 用户触发命令时自动检测是否存在历史分析记录
- 若不存在历史记录：自动进入全量解析模式
- 若存在历史记录：
  1. 获取当前commit hash，检查是否在历史git_commits列表中
  2. 若存在匹配记录：直接使用该commit作为基准计算差异
  3. 若不存在匹配记录：调用git API查找当前commit与历史commit列表的最近共同祖先，作为基准计算差异
  4. 仅当找不到任何关联的历史commit时，才触发全量解析
  5. 提取所有变更文件（新增/修改/删除），仅重新解析变更文件及其所属的上级目录
  6. 对受影响的目录分析结果进行更新，同步更新项目整体结构图和总结文档
- 若当前项目不是git项目：每次触发都默认进入全量解析模式

#### 3.4.3 冲突处理

- 若本地文件存在未提交的变更：提示用户`检测到未提交的变更，是否继续增量解析？`，用户确认后继续
- 若当前分支与上次解析分支不一致：优先查找共同祖先commit做增量解析，仅当无关联记录时提示用户`检测到分支切换，将进行全量解析`

## 4. 存储设计规范

### 4.1 存储位置

遵循OpenCode本地存储规范：

- 存储根目录：`~/.local/share/opencode/project/{项目slug}/storage/code-analysis/`
- 项目slug生成规则：`{仓库所有者}-{仓库名称}`（git项目）或`{目录名称}-{路径hash}`（非git项目）

### 4.2 存储目录结构

与原项目目录结构完全保持一致：

```
code-analysis/
├── .analysis_metadata.json      # 解析元数据（含最近50次commit记录）
├── ANALYSIS_MODIFICATION_LOG.md # 修正记录日志
├── PROJECT_SUMMARY.md           # 项目整体总结、结构图、流程图
├── src/
│   ├── index.md                 # src目录分析结果
│   ├── components/
│   │   ├── Button.md            # Button.tsx文件分析结果
│   │   └── Input.md             # Input.tsx文件分析结果
│   └── utils/
│       └── index.md             # utils目录分析结果
└── tests/
    └── index.md                 # tests目录分析结果
```

### 4.3 文件命名规则

- 目录：`{目录名}.md`
- 文件：`{原文件名（不含后缀）}.md`
- 特殊文件：统一放在存储根目录，命名采用大写蛇形

## 5. 非功能需求

### 5.1 性能要求

- 单文件解析耗时不超过2s
- 1000个文件的项目全量解析耗时不超过5分钟（基于默认并发数配置）
- 增量解析耗时与变更文件数量成正比，单变更文件处理不超过3s
- 并行解析资源占用不超过系统CPU的70%，内存占用不超过500MB

### 5.2 兼容性要求

- 支持OpenCode v1.0+版本
- 支持Windows/macOS/Linux全平台
- 支持所有主流编程语言的代码解析

### 5.3 可靠性要求

- 解析过程异常中断后，下次启动可从断点继续（已完成的目录解析结果不会丢失）
- 历史分析结果不会被覆盖，可通过commit hash回溯任意历史版本的解析结果
- 所有生成的Markdown文件无语法错误，可直接渲染
- 跨分支增量解析准确率不低于95%，仅5%以下场景需要回退全量解析

## 6. 开发约束

### 6.1 技术栈约束

- 采用TypeScript开发，符合OpenCode插件开发规范
- 依赖包通过插件目录下的`package.json`管理
- 使用OpenCode SDK提供的API进行git操作和文件访问
- 并行任务采用worker pool实现，使用异步IO处理文件读写

### 6.2 规范约束

- 日志采用`client.app.log()`接口输出，不使用`console.log`
- 所有工具参数采用Zod schema验证
- 遵循OpenCode事件驱动架构，通过`file.watcher.updated`事件监听文件变更实现实时更新
- 并发数控制采用动态调整策略，根据系统负载自动限流

### 6.3 安全约束

- 不读取、不上传任何代码内容到第三方服务，所有分析在本地完成
- 不修改原项目的任何文件，所有结果写入专用存储目录
- 避免解析敏感文件（`.env`、`credentials.json`等），自动加入忽略列表

## 8. V2.0 独立CLI版本需求变更

### 8.1 变更概述

#### 8.1.1 背景

原V1.0版本作为OpenCode平台内置插件运行，完全依赖OpenCode SDK和平台运行时环境，无法独立使用。为满足用户本地终端直接使用的需求，V2.0版本调整为完全独立的终端命令行工具，移除所有OpenCode相关依赖，不再与OpenCode平台有任何关联。

#### 8.1.2 变更目标

- 支持在任意本地环境终端直接调用所有功能
- 保留V1.0版本所有核心代码理解、解析、查询能力
- 新增终端交互、配置管理、多格式输出等CLI专属能力
- 实现跨Windows/macOS/Linux全平台兼容

### 8.2 影响范围

本次变更影响以下原有功能模块：


| 模块名称   | 原有依赖                             | 变更影响                          |
| ------ | -------------------------------- | ----------------------------- |
| 命令入口模块 | 依赖OpenCode SDK的`tool()`接口注册命令    | 完全重构为原生CLI命令入口                |
| 参数解析模块 | 依赖OpenCode平台参数透传                 | 新增原生CLI参数解析、校验、自动补全能力         |
| 日志输出模块 | 依赖OpenCode平台`client.app.log()`接口 | 重构为原生终端彩色日志输出                 |
| 结果输出模块 | 依赖OpenCode平台结果上报规范               | 新增多格式本地输出能力（文本/JSON/Markdown） |
| 配置管理模块 | 无独立配置体系，依赖平台配置                   | 新增三级配置体系（命令行参数>环境变量>本地配置文件）   |
| 存储模块   | 依赖OpenCode本地存储规范                 | 调整为自定义本地存储路径，支持用户自定义输出目录      |
| 交互模块   | 依赖OpenCode平台对话框交互                | 新增终端交互能力（进度条、确认提示、选择交互等）      |


### 8.3 新需求明细

#### 8.3.1 CLI命令行入口设计

- 主命令名称：`code-analyze`（短别名：`ca`）
- 子命令划分：

  | 子命令       | 功能说明                              |
  | --------- | --------------------------------- |
  | `analyze` | 执行项目代码解析，对应原有`/analyze-project`命令 |
  | `query`   | 查询解析结果，对应原有`project-code-query`技能 |
  | `config`  | 管理本地配置，查看/修改默认配置                  |

- 全局公共参数：

  | 参数                      | 说明                                 | 默认值                                  |
  | ----------------------- | ---------------------------------- | ------------------------------------ |
  | `-h, --help`            | 显示帮助信息                             | -                                    |
  | `-v, --version`         | 显示版本号                              | -                                    |
  | `-c, --config <path>`   | 指定配置文件路径                           | `~/.config/code-analyze/config.yaml` |
  | `-o, --output <format>` | 输出格式：`text`/`json`/`markdown`      | `text`                               |
  | `--log-level <level>`   | 日志级别：`debug`/`info`/`warn`/`error` | `info`                               |

- 自动生成命令帮助信息、参数示例，支持tab自动补全

#### 8.3.2 依赖清理

- 移除所有`@opencode/sdk`相关依赖包和导入
- 移除所有OpenCode平台专属API调用（`tool()`、`client.app.log()`等）
- 移除所有与OpenCode生态相关的逻辑（事件监听、平台权限校验等）

#### 8.3.3 终端交互优化

- 进度条展示：解析过程中实时显示进度条，包含已完成文件数、总文件数、剩余时间、当前处理文件
- 彩色日志输出：错误（红）、警告（黄）、成功（绿）、提示（蓝）分颜色高亮显示
- 分级日志系统：
  - 支持4种日志级别：`debug`/`info`/`warn`/`error`，默认级别为`info`
  - 级别可通过全局参数`--log-level <level>`、环境变量`CODE_ANALYZE_LOG_LEVEL`或配置文件中`global.log_level`配置
  - 日志内容说明：
    - `error`：仅打印致命错误信息，程序无法继续执行的场景
    - `warn`：打印警告信息，不影响主流程但需要用户注意的场景
    - `info`（默认）：打印关键运行信息，包括解析开始/完成提示、进度信息、操作结果等核心日志
    - `debug`：打印详细调试信息，包括每个文件的解析过程、参数详情、内部执行步骤、错误堆栈等，用于开发调试和问题排查
  - 日志输出到标准终端stdout/stderr，可通过管道重定向到文件
- 友好错误提示：错误信息包含错误原因、解决建议、错误码
- 交互确认：高危操作（如覆盖已有解析结果、全量重解析）弹出终端确认提示，用户确认后继续执行

#### 8.3.4 配置体系

- 三级配置优先级（从高到低）：命令行参数 > 环境变量（`CODE_ANALYZE_*`前缀） > 本地配置文件（`~/.config/code-analyze/config.yaml`）
- 配置文件结构：
  ```yaml
  # 全局配置
  global:
    log_level: info
    output_format: text
    auto_confirm: false
    output_dir: ./.code-analyze-result

  # analyze命令默认配置
  analyze:
    default_mode: auto
    default_concurrency: 4
    default_depth: -1
    exclude_patterns: ["node_modules", ".git", "dist", "build"]

  # query命令默认配置
  query:
    default_type: summary
  ```

#### 8.3.5 结果输出

- 支持三种输出格式：
  - `text`：纯文本格式，适合终端直接查看
  - `json`：结构化JSON格式，适合脚本二次处理
  - `markdown`：Markdown格式，适合生成报告文档
- 支持自定义输出目录，默认存储在被解析项目根目录下的`.code-analyze-result`目录
- 支持直接输出到文件（`code-analyze analyze ./src -o markdown > report.md`）

#### 8.3.6 兼容性

- 支持Windows 10+/macOS 12+/Ubuntu 20.04+全平台终端运行
- 支持Node.js 16.0+版本
- 命令参数尽量与V1.0保持一致，降低老用户迁移成本

- 原存储规范中OpenCode专属存储路径调整为自定义本地存储路径

## 9. V2.1 LLM原生解析需求变更

### 9.1 变更概述

#### 9.1.1 背景

原V2.0版本采用静态代码解析逻辑实现代码分析，存在支持语言有限、解析规则需要人工维护、复杂逻辑理解能力不足等问题。为实现全语言代码解析能力、提升复杂代码逻辑理解准确率，V2.1版本将原有静态解析逻辑完全替换为LLM原生解析，依托LLM的代码理解能力实现任意格式文本代码文件的解析。

#### 9.1.2 变更目标

- 移除所有静态代码解析逻辑，完全基于LLM实现代码文件解析
- 保留原有所有输出规范、解析架构、功能能力不变
- 支持用户全量自定义LLM调用配置
- 实现大文件自动分片解析与合并能力
- 优化缓存策略降低LLM调用成本

### 9.2 影响范围

本次变更影响以下原有功能模块：


| 模块名称    | 原有实现           | 变更影响               |
| ------- | -------------- | ------------------ |
| 代码解析模块  | 基于静态语法分析实现     | 完全重构为LLM调用解析逻辑     |
| 配置管理模块  | 无LLM相关配置       | 新增LLM全量配置项         |
| 大文件处理模块 | 基于静态分析的大文件拆分逻辑 | 重构为LLM分片解析+结果合并逻辑  |
| 缓存模块    | 原有文件修改时间缓存     | 新增基于文件哈希的解析结果缓存    |
| 错误处理模块  | 静态解析错误处理逻辑     | 新增LLM调用失败的交互处理逻辑   |
| 安全约束模块  | 强制本地解析不上传代码    | 移除本地上传约束，支持远程LLM调用 |


### 9.3 新需求明细

#### 9.3.1 LLM解析核心逻辑

- 解析方式：每个文本代码文件直接发送到LLM进行解析，由LLM完成所有文件级分析工作
- 语言支持：LLM原生支持任意编程语言的文本代码文件解析，无文件后缀限制
- 输出要求：严格要求LLM按照原需求定义的输出格式返回结果，包含文件基本信息、核心作用、类定义、Mermaid类图/时序图等固定内容
- 架构保留：原有两轮解析架构完全保留，第一轮自底向上解析、第二轮自顶向下校验修正均调用LLM完成

#### 9.3.2 LLM配置管理

- 配置项：支持用户配置所有LLM调用相关参数，包括：模型名称、API服务地址、API密钥、temperature、max_tokens、timeout、代理配置等
- 配置方式：支持通过命令行参数、环境变量、配置文件三种方式配置LLM相关参数
- 配置优先级：遵循原三级配置优先级规则（命令行参数 > 环境变量 > 本地配置文件）

#### 9.3.3 大文件处理逻辑

- 分片规则：文件长度超过配置的LLM上下文窗口限制时，自动按代码语义边界拆分为多个分片
- 分片解析：每个分片单独调用LLM完成解析，生成分片级解析结果
- 结果合并：所有分片解析完成后，将全部分片解析结果整合为一个请求发送给LLM，生成该文件的最终统一解析结果

#### 9.3.4 缓存策略

- 缓存规则：按文件内容哈希值缓存解析结果，文件内容未发生变更时直接复用缓存结果
- 缓存有效期：默认永久有效，支持用户手动清除缓存
- 缓存存储：缓存结果存储在解析结果目录下的`.cache`目录中

#### 9.3.5 错误处理逻辑

- LLM调用失败（网络超时、限流、返回格式错误等）时，立即暂停解析流程
- 终端提示用户调用失败信息，提供三个选项：[1] 重试调用 [2] 跳过当前文件继续解析 [3] 终止整个解析流程
- 根据用户选择执行对应操作

#### 9.3.6 安全约束调整

- 移除原需求中"不读取、不上传任何代码内容到第三方服务，所有分析在本地完成"的约束
- 使用远程LLM时，所有解析的代码文件内容都会上传到配置的LLM服务，相关风险由用户自行承担
- 解析过程中明确提示用户代码将会上传到LLM服务

#### 9.3.7 解析范围限制

- 仅支持文本格式代码文件解析，不支持非文本格式的多模态文件（流程图、设计文档、图片等）

#### 9.3.8 结果校验策略

- 完全信任LLM返回的解析结果，不做额外的一致性校验
- 不限制LLM的输出内容，只要符合原格式要求即可

#### 9.3.9 成本控制

- 不做LLM调用次数、Token使用量的限制和拦截
- 仅在解析完成后输出本次解析的总调用次数、总Token使用量统计信息供用户参考

### 9.4 原有需求调整说明

- 原需求中所有与静态代码解析相关的技术实现约束全部移除
- 原安全约束中关于代码不上传的条款已删除
- 原性能要求中解析耗时指标调整为：单文件解析耗时包含LLM调用耗时，具体耗时取决于用户配置的LLM服务响应速度

### 9.5 需求优先级


| 需求模块           | 优先级 | 版本   |
| -------------- | --- | ---- |
| LLM解析核心逻辑+配置管理 | P0  | v2.1 |
| 大文件分片解析与合并     | P0  | v2.1 |
| 解析缓存机制         | P1  | v2.1 |
| LLM调用错误交互处理    | P1  | v2.1 |

## 10. V2.2 解析结构与增量优化需求变更

### 10.1 变更概述

#### 10.1.1 背景

在V2.1完全切换为LLM原生解析的基础上，现有实现仍存在以下问题：

- 进度展示不够精确，无法直观反映「总对象数」与「已完成对象数」；
- 增量解析粒度较粗，主要基于仓库级git commit，缺少对单文件级别的精细缓存控制；
- 解析对象范围过宽，包含大量配置/文档/二进制文件，增加不必要的LLM调用成本；
- 文件与目录的解析结果结构不够统一、可控，对后续程序消费支持不足；
- 大文件/大目录场景下的分片策略需要进一步明确；
- 根目录存在特殊处理逻辑，整体流程不够统一；
- 自顶向下第二轮解析复杂度高、价值有限。

为解决上述问题，V2.2在不改变CLI整体交互模式的前提下，对进度展示、增量解析策略、文件过滤规则、LLM调用协议、分片逻辑和目录处理方式进行优化和约束。

#### 10.1.2 变更目标

- 明确「对象级」进度统计规范，统一以「文件+目录」为基本计数单位；
- 在保留仓库级增量机制的前提下，引入「文件级git commit id + 文件哈希」双重校验缓存策略；
- 严格限定解析范围，仅处理代码类文本文件，排除二进制、配置、文档类文件；
- 重构文件/目录解析结果结构，统一由LLM输出JSON格式内容，再由程序写入Markdown结果；
- 明确大文件与大目录的硬切分分片策略及对应的LLM交互流程；
- 取消根目录特殊处理逻辑，使其与普通目录完全一致；
- 取消第二轮自顶向下解析，仅保留第一轮自底向上解析。

### 10.2 进度展示规范

#### 10.2.1 统计对象定义

- 解析过程中的「对象」统一定义为：
  - 将要尝试处理的所有「文本代码文件」；
  - 将要生成解析结果的所有「目录」。
- 总对象数 = 文本代码文件数 + 目录数。
- 其中「文本代码文件」的定义见10.4节。

#### 10.2.2 进度展示内容

- 在终端实时输出的进度信息中，至少包含以下内容：
  - 已完成对象数 / 总对象数（对象数=文件数+目录数）；
  - 其中已完成文件数、已完成目录数（可作为补充信息输出）；
  - 当前正在处理的文件或目录路径列表（可截断显示）。
- 对象被视为「已完成」的条件：
  - 文件对象：
    - 已完成一次「解析流程」或
    - 被判定为「无需重新解析」（命中git+哈希缓存）或
    - 被判定为「非解析范围对象」（如非文本文件、被过滤文件）；
  - 目录对象：
    - 已完成目录级汇总逻辑并生成对应Markdown结果。

#### 10.2.3 进度与并发

- 进度更新与worker pool统一集成：
  - 每当一个对象（文件/目录）处理完毕（无论是实际解析、缓存命中跳过还是过滤跳过），均触发一次进度更新；
  - 进度条应真实反映所有对象的处理状态，不出现长时间不动的假进度。
- 支持与日志级别联动：
  - 在`info`级别输出简要进度信息；
  - 在`debug`级别可输出更详细的当前任务队列、正在处理对象明细等信息。

### 10.3 增量解析与缓存策略

V2.2在V1.0/V2.1的仓库级增量解析基础上，引入「文件级git commit id + 文件哈希」的双重缓存控制。整体原则：

- 优先使用git信息判断是否有必要进入候选变更集；
- 在候选集内，再结合文件级git commit id与文件内容哈希判断是否需要重新解析；
- 对非git目录或git不可用场景，退化为基于文件哈希的缓存策略。

#### 10.3.1 仓库级增量逻辑（保留）

- 继续沿用3.4节的仓库级增量解析逻辑：
  - 维护`.analysis_metadata.json`中最近解析的仓库级`git_commits`列表；
  - 根据当前commit及历史记录/共同祖先计算变更文件集合；
  - 若无法找到有效基准commit，则退回全量解析。

#### 10.3.2 文件级git commit id

- 对于处于git仓库内的每个文件，定义其「文件级git commit id」为：
  - 使用`git log -n 1 --pretty=format:%H -- <file>`获取的该文件最近一次被修改的commit hash；
  - 若该命令无法返回结果（例如文件为新建但未提交），则视为「无文件级git commit id」。
- 文件解析结果Markdown中须记录以下元信息：
  - `file_git_commit_id`：文件级git commit id；
  - `is_dirty_when_analyzed`：布尔值，表示解析当时文件是否处于「有未提交修改」状态；
  - `file_hash_when_analyzed`：解析时文件内容的哈希值（用于后续无git场景或dirty场景对比）。

#### 10.3.3 文件哈希

- 文件哈希用于：
  - 非git目录或git不可用场景的变更检测；
  - 有未提交修改（dirty）时的状态跟踪；
  - 在特定场景下移除/恢复「未提交修改」标记。
- 哈希计算要求：
  - 对文件完整内容计算；
  - 算法可采用通用哈希算法（如SHA-256），但文档中不限定具体算法实现细节。

#### 10.3.4 解析与跳过决策逻辑

当决定是否对某个文件重新解析时，遵循以下规则：

1. 首先通过仓库级增量逻辑确定候选文件集合：
  - 若当前项目为git仓库且存在有效历史记录：
    - 使用仓库级增量逻辑计算变更文件集合；
    - 仅对变更文件及其上级目录进入后续判定；
  - 若非git仓库或无法获取有效历史记录：
    - 视为全量候选集合。
2. 对候选文件，按以下优先级判断是否需要重新解析：
  - 若存在对应的历史解析结果Markdown：
    - 读取其中记录的`file_git_commit_id`、`is_dirty_when_analyzed`、`file_hash_when_analyzed`；
    - 再结合当前文件状态进行判断：
  1. 当前文件有有效git commit id（已提交）：
    - 若当前`file_git_commit_id`与历史记录中的`file_git_commit_id`相同，且历史解析时`is_dirty_when_analyzed`为`false`：
      - **视为缓存命中，直接跳过解析**；
    - 若当前`file_git_commit_id`与历史不同：
      - **需要重新解析**；
  2. 当前文件处于「有未提交修改（dirty）」状态：
    - 若历史记录中`is_dirty_when_analyzed`为`true`，且本次计算出的`file_hash`与`file_hash_when_analyzed`相同：
      - **视为缓存命中，直接跳过解析**；
    - 否则：
      - **需要重新解析**。
      历史解析结果Markdown（首次或路径变化等）：
      需要重新解析**。
3. 在重新解析完成后，更新Markdown中的元信息：
  - 若解析时文件为已提交状态：
    - `file_git_commit_id`更新为当前文件级git commit id；
    - `is_dirty_when_analyzed=false`；
    - `file_hash_when_analyzed`更新为当前文件内容哈希。
  - 若解析时文件为dirty状态：
    - `file_git_commit_id`可以为空或保持历史值（若存在）；
    - `is_dirty_when_analyzed=true`；
    - `file_hash_when_analyzed`更新为当前文件内容哈希。
4. 后续解析时，对于历史上在dirty状态下解析过的文件：
  - 若本次文件已处于「提交完成」状态，且当前文件内容哈希与`file_hash_when_analyzed`相同：
    - 视为「内容未变，仅状态从dirty变为已提交」；
    - 允许在本次解析中不重新调用LLM，仅更新Markdown中的元信息：
      - 将`is_dirty_when_analyzed`更新为`false`；
      - 补写当前有效的`file_git_commit_id`。

#### 10.3.5 目录级联更新

- 当以下任一情况发生时，目录级解析结果必须重新生成：
  - 目录下任意子文件的解析结果Markdown发生变化（包括内容变更或仅元信息变更）；
  - 目录下任意子目录的解析结果Markdown发生变化；
  - 目录下子文件或子目录的集合发生变更（新增/删除/重命名）；
  - 当前目录自身的基本信息变更（例如目录名变化等）。
- 重算目录时不再做任何仓库级/文件级缓存判断，始终以当前子项的最新解析结果为准生成目录Markdown。

### 10.4 解析范围与文件过滤规则

#### 10.4.1 文本文件判定

- 仅解析「文本格式的代码文件」，不解析任何二进制文件。
- 文本/二进制判定规则：
  - 可以使用第三方库（如`istextorbinary`）或等价逻辑，对文件内容进行探测；
  - 对典型二进制扩展名（图片、视频、字体、压缩包等）可直接视为二进制，无需内容探测；
  - 对无法确定的文件，可通过读取部分字节检查是否包含大量不可见控制字符、NUL字节等，若超过阈值则视为二进制。
- 被判定为二进制的文件：
  - 不进行LLM解析；
  - 不生成任何对应的Markdown结果文件；
  - 在所在目录的Markdown中也不列入文件描述列表。

#### 10.4.2 黑名单过滤规则

- 在文本判定通过的前提下，仍需对以下类别的文件进行过滤，不参与解析：
  - 配置文件（例如常见的`.yml`/`.yaml`、`.ini`、`.cfg`、部分`.json`等，具体规则由实现维护黑名单列表）；
  - README类文件及其他文档文件（例如`README`*、`*.md`、`*.txt`等，具体规则由实现维护黑名单列表）；
  - 其他不希望纳入代码解析范围的文件类型或路径（例如`docs/`目录下的文档等）。
- 黑名单方案要求：
  - 使用「黑名单」策略，不采用「白名单」；
  - 黑名单可基于：
    - 文件扩展名模式；
    - 文件名模式；
    - 路径模式（目录/子目录）。
- 被黑名单过滤的文件：
  - 不进行LLM解析；
  - 不生成任何对应Markdown结果；
  - 不计入目录Markdown的文件描述列表。

### 10.5 文件解析结果结构与LLM调用协议

V2.2对文件级解析结果的结构和LLM调用流程进行统一约束，目标是：

- 所有由LLM生成的内容在一次文件解析流程中以JSON格式输出，便于程序解析；
- 最终写入磁盘的仍为Markdown格式文件，但内部结构清晰、可预测；
- 对大文件分片场景也遵循相同的JSON结构。

#### 10.5.1 文件Markdown总体结构

- 每个文件对应一个Markdown结果文件，基本包含以下部分：
  - 基本信息（由程序直接生成）：
    - 文件名；
    - 语言（若有）；
    - 文件行数；
    - 最后解析时间；
    - `file_git_commit_id`；
    - `is_dirty_when_analyzed`；
    - `file_hash_when_analyzed`；
  - LLM解析结果（由程序根据LLM的JSON输出转写）：
    - 概述（100字以内）；
    - 功能描述（200字以内）；
    - 所有类定义；
    - 全局变量定义；
    - 全局函数定义。
- Markdown内部可采用合适的标题层级和代码块展示JSON详情，但最终对外接口仍以Markdown文件为准。

#### 10.5.2 LLM输出JSON结构要求

- 对于单个文件的一次完整解析，LLM需输出一个JSON对象，包含至少以下字段：
  - `summary`: string  
    - 文件整体概述，100字以内；
  - `description`: string  
    - 文件功能的详细描述，200字以内；
  - `classes`: ClassDefinition[]  
  - `global_variables`: GlobalVariableDefinition[]  
  - `global_functions`: GlobalFunctionDefinition[]
- 其中：
  - `ClassDefinition`：
    - `name`: string
    - `description`: string（200字以内）
    - `fields`: FieldDefinition[]
    - `methods`: MethodDefinition[]
  - `FieldDefinition`：
    - `visibility`: string（如`public`/`private`/`protected`等）
    - `type`: string
    - `name`: string
    - `description`: string（一句话描述）
  - `MethodDefinition`：
    - `visibility`: string
    - `return_type`: string
    - `name`: string
    - `description`: string（200字以内）
  - `GlobalVariableDefinition`：
    - `name`: string
    - `type`: string
    - `description`: string（一句话描述）
  - `GlobalFunctionDefinition`：
    - `name`: string
    - `return_type`: string
    - `description`: string（200字以内）
- 若某类信息不存在（例如文件中无类定义），对应字段必须仍然存在，但取空数组`[]`或空字符串`""`，不允许省略字段。

#### 10.5.3 LLM调用次数与顺序（单文件、非分片）

- 对于不需要分片的普通文件，LLM调用流程要求：
  - 使用单一会话上下文完成三次交互，顺序固定且每次只生成一个内容：
    1. **第一次调用**：
      - 输入：完整文件内容 + 明确的提问；
      - 任务：仅提取「所有类定义、全局变量、全局函数」的结构信息，并以JSON形式返回（不包含功能描述和概述）；
    2. **第二次调用**：
      - 在同一会话上下文中继续；
      - 输入：引用前一次返回的结构信息，请求生成文件的「功能描述」；
      - 任务：仅生成`description`字段的内容；
    3. **第三次调用**：
      - 仍在同一会话上下文中；
      - 输入：基于前两次的信息，请求生成文件的「概述」；
      - 任务：仅生成`summary`字段的内容。
- 程序端责任：
  - 在三次调用结束后，将三部分信息组装为符合10.5.2所述结构的完整JSON对象；
  - 再将该JSON对象转换为Markdown格式写入结果文件。

#### 10.5.4 LLM输出错误重试

- 当某次调用的输出无法成功解析为期望的JSON格式时：
  - 程序应捕获解析异常，并自动对该次调用重试一次；
  - 重试时可在提示中明确说明「上一次输出格式错误，请严格按照给定JSON结构输出」；
  - 若重试仍然失败：
    - 可按照现有错误处理策略执行（例如提示用户选择重试/跳过/终止），具体交互逻辑与9.3.5保持一致。

### 10.6 目录解析结果结构与LLM调用协议

#### 10.6.1 目录Markdown总体结构

- 每个目录对应一个Markdown结果文件，包含以下部分：
  - 基本信息（程序生成）：
    - 目录名；
    - 子目录数量；
    - 文件数量（仅统计纳入解析范围的文本代码文件）；
    - 最后解析时间；
  - LLM生成内容：
    - 目录功能描述（200字以内）；
    - 目录概述（100字以内）；
  - 子目录描述列表（程序生成）：
    - 每个子目录一行，格式：`<目录名:概述>`；
    - 概述从对应子目录Markdown的概述部分复制；
  - 文件描述列表（程序生成）：
    - 每个文件一行，格式：`<文件名:概述>`；
    - 概述从对应文件Markdown的概述部分复制；
  - 仅展示当前目录的直接子项，不递归展示更深层级。

#### 10.6.2 目录LLM输入内容

- 目录级LLM调用使用「精简子项信息」作为输入，而非完整子项Markdown内容：
  - 对于每个直接子目录，提供：
    - 子目录名；
    - 子目录概述；
    - 子目录功能描述；
  - 对于每个直接子文件，提供：
    - 文件名；
    - 文件概述；
    - 文件功能描述；
  - 这些信息可通过解析对应子项的Markdown结果获得。

#### 10.6.3 LLM调用次数与顺序（目录、非分组）

- 对于不需要分组的普通目录，LLM调用流程：
  - 使用单一会话上下文完成两次交互，顺序固定且每次只生成一个内容：
    1. **第一次调用**：
      - 输入：当前目录下所有直接子目录和子文件的精简信息；
      - 任务：生成整个目录的「功能描述」（200字以内）；
    2. **第二次调用**：
      - 在同一会话中；
      - 输入：基于第一次的功能描述和子项精简信息；
      - 任务：生成整个目录的「概述」（100字以内）。
- LLM返回格式：
  - 可为简单自然语言字符串，不强制JSON格式；
  - 由程序将返回内容嵌入目录Markdown对应位置。

### 10.7 大文件与大目录的分片策略

#### 10.7.1 文件分片（大文件）

- 当单个文件内容长度超过配置的LLM上下文限制阈值时，必须进行硬切分分片处理：
  - 不使用复杂的语义边界切分，仅按字节/字符/行数硬切分；
  - 目标是保证每个分片在附加必要提示信息后，仍然在LLM上下文上限之内。
- 分片阶段LLM调用：
  - 对每个分片，仅进行一次调用；
  - 任务：仅提取该分片中可确定的「类定义、全局变量、全局函数」信息，并以JSON形式返回；
  - 不在分片阶段生成功能描述和概述。
- 合并阶段：
  - 所有分片解析完成后，程序将所有分片的JSON结果合并为一个中间结构（字段结构与10.5.2一致，可能存在重复/不完整信息）；
  - 启动一个新的LLM会话，将该中间结构作为输入；
  - 在该会话中按10.5.3所述三步流程完成完整文件解析：
    1. 基于合并后的结构重新整理「所有类定义、全局变量、全局函数」；
    2. 生成文件功能描述；
    3. 生成文件概述。
- 提示要求：
  - 在分片调用的提示中，明确告知LLM「你看到的是文件的一部分，可能不完整，请只根据当前可确定的内容输出结构化信息，不要猜测分片外的内容」；
  - 在合并阶段的提示中，明确告知LLM「你看到的是多个分片解析结果的合并视图，请统一、去重并输出完整的结构信息」。

#### 10.7.2 目录分组（大目录）

- 当某个目录下的子文件/子目录数量过多，以至于其精简信息整体超出LLM上下文限制时，需要对精简信息进行分组处理：
  - 按子项数量顺序或其他合理方式划分为若干组；
  - 每组的精简信息大小控制在LLM上下文限制之内。
- 分组阶段LLM调用：
  - 对每一组子项：
    - 启动一个独立LLM会话；
    - 输入该组内所有子目录/子文件的精简信息；
    - 任务：仅生成「该组视角下的目录功能描述」，以自然语言形式返回。
- 合并阶段：
  - 所有分组功能描述生成完毕后：
    - 启动一个新的LLM会话；
    - 将所有分组的功能描述作为输入；
    - 在该会话中按10.6.3所述两步流程完成完整目录解析：
      1. 基于所有分组功能描述生成最终的目录功能描述；
      2. 在此基础上生成最终的目录概述。

### 10.8 根目录处理与整体架构调整

#### 10.8.1 根目录与普通目录统一

- 取消根目录的任何特殊处理逻辑：
  - 根目录与普通目录在解析流程、LLM调用次数与顺序、Markdown结构上完全一致；
  - 不再产生单独的`PROJECT_SUMMARY.md`文件。
- 项目整体总结：
  - 由根目录Markdown中的「概述」和「功能描述」自然承担；
  - 若用户需要查看项目整体总结，可直接查询根目录对应的结果。

#### 10.8.2 第二轮自顶向下解析取消

- 原2.2.2节定义的「第二轮：自顶向下校验修正」在V2.2中完全取消：
  - 不再执行基于父目录结果对下层结果的自动修正逻辑；
  - 不再生成`ANALYSIS_MODIFICATION_LOG.md`文件；
  - 所有文件和目录的解析结果仅由第一轮自底向上的解析流程产生。
- 全局结构图、流程图等原有全局性内容：
  - 若后续仍需保留，可在第一轮自底向上的解析结果基础上另行定义生成逻辑；
  - 本次V2.2变更不再新增该类逻辑，亦不在需求中约束其实现细节。

### 10.9 解析结构与LLM调用协议表述修正

本节对 10.5、10.6、10.7 中与「分次调用、程序组装」不一致或模糊的表述进行统一澄清，不改变既有流程与字段定义，仅明确：**文件与目录解析均为多次对 LLM 提问，每次 LLM 返回一段结果（如一次一个 JSON 或部分字段），由程序合并/组装成最终结构**。

#### 10.9.1 文件解析：目标表述与 10.5.2 的定位

- **10.5 节开头目标句（原易误解为「一次 LLM 输出」）**  
应理解为：在一次**文件解析流程**中，通过**多次** LLM 调用得到多段 JSON/字段，由程序组装为一份符合 10.5.2 的完整 JSON，再转换为 Markdown。其中「一次」指「一次文件解析流程」，而非「一次 LLM 调用」。
- **10.5.2 的定位**  
10.5.2 描述的是**程序组装后的、对外/落盘使用的完整 JSON 结构**。该结构**并非**由 LLM 单次输出，而是由 10.5.3 中多次 LLM 调用的结果在程序端合并而成。实现与文档中应明确区分「LLM 单次返回形态」与「程序组装后的目标结构」。
- **各次 LLM 调用的返回形态（与 10.5.3 对应）**  
单文件、非分片场景下，三次调用的返回约定如下，便于程序解析与组装：
  - **第一次调用**：仅返回「类定义、全局变量、全局函数」的结构化信息，JSON 形态与 10.5.2 中的 `classes`、`global_variables`、`global_functions` 一致；不包含 `summary`、`description`。
  - **第二次调用**：仅返回「功能描述」；可为 `{ "description": "..." }` 或约定格式的纯字符串，由程序填入 10.5.2 的 `description` 字段。
  - **第三次调用**：仅返回「概述」；可为 `{ "summary": "..." }` 或约定格式的纯字符串，由程序填入 10.5.2 的 `summary` 字段。  
  程序在三次调用结束后，将上述三部分组装为 10.5.2 所述的完整 JSON，再转为 Markdown 写入结果文件。

#### 10.9.2 文件解析：错误重试粒度（10.5.4）

- 当**某一次**调用的输出无法成功解析为期望的 JSON 或约定格式时，仅对该**当次**调用进行重试，不重做此前已成功的步骤。
- 重试时可在提示中说明「上一次输出格式错误，请严格按照给定结构输出」；若重试仍失败，按 9.3.5 等既有错误处理策略执行。

#### 10.9.3 目录解析：完整输出结构与 10.6.3 的对应关系

- **目录的「完整输出结构」**  
目录解析的最终结果在程序侧对应 10.6.1 的 Markdown 结构，其中由 LLM 生成的两部分为：
  - 目录功能描述（200 字以内）；
  - 目录概述（100 字以内）。  
  该「完整输出」**由两次 LLM 调用的结果在程序端组合而成**，并非 LLM 一次输出。
- **各次调用的返回形态与组装方式（对应 10.6.3）**  
  - **第一次调用**：输入为当前目录下所有直接子目录和子文件的精简信息；任务为生成整个目录的「功能描述」；返回可为自然语言或 `{ "description": "..." }` 等约定格式，由程序写入 10.6.1 中「目录功能描述」位置。
  - **第二次调用**：在同一会话中，基于第一次的功能描述与子项精简信息；任务为生成整个目录的「概述」；返回可为自然语言或 `{ "summary": "..." }` 等约定格式，由程序写入 10.6.1 中「目录概述」位置。  
  是否强制 JSON 格式可由实现约定，以利于解析与错误重试；若采用自然语言，需在 10.6.3 中明确程序如何从返回文本中抽取并嵌入对应字段。

#### 10.9.4 大文件分片合并阶段与 10.5.3 的一致性（10.7.1）

- 分片阶段对每个分片仅做一次调用，提取该分片内的结构信息并以 JSON 返回；合并阶段得到「多分片合并后的中间结构」后，**仍按 10.5.3 的三步流程**在程序侧执行：
  1. 基于合并后的结构，通过一次 LLM 调用重新整理「所有类定义、全局变量、全局函数」；
  2. 再通过一次调用生成文件功能描述；
  3. 再通过一次调用生成文件概述。
  即：合并后的完整解析仍为**三次** LLM 调用，每次的输入/输出形态与单文件非分片时一致，仅输入从「单次结构 JSON」变为「多分片合并后的中间结构」。10.7.1 中「按 10.5.3 所述三步流程」应按此理解。

## 11. V2.3 路径查询 Skill 生成与黑名单优化需求变更

### 11.1 变更概述

#### 11.1.1 背景

V2.2 完成了解析结构统一和增量优化后，系统已能可靠地生成结构化的 Markdown 分析结果。但当前存在以下不足：

- 解析结果仅支持人工浏览，缺乏程序化的路径查询能力，无法快速根据源码路径定位到对应的分析结果文件；
- LLM（如 OpenCode、Cursor、Claude Code、Codex 等 AI 编程工具）在编码过程中无法自动利用已有的解析结果，每次需要理解代码时仍需从零开始；
- 文件过滤规则采用硬编码的简单模式列表（`IGNORED_PATTERNS`），不支持项目级自定义，也不兼容 `.gitignore` 语法，灵活性不足。

为解决上述问题，V2.3 引入集中式索引文件、路径查询脚本、Agent Skill 自动生成机制以及基于 `.gitignore` 语法的多层黑名单过滤体系。

#### 11.1.2 变更目标

- 在解析阶段生成集中式索引文件（`analysis-index.json`），建立「源码绝对路径 → 分析结果 Markdown 绝对路径」的映射关系；
- 实现路径查询功能：输入任意文件或目录的绝对路径，返回对应分析结果 Markdown 文件的绝对路径（不存在则返回 `N/A`）；
- 将路径查询功能封装为符合 [Agent Skills 开放标准](https://agentskills.io/) 的 Skill，在解析完成后自动部署到被解析项目中，使主流 AI 编程工具可直接发现并使用；
- 支持用户选择为哪些 AI 工具生成 Skill，默认工具列表由配置文件决定；
- 重构文件过滤规则，废弃旧版 `exclude_patterns`/`IGNORED_PATTERNS`，改用遵循 `.gitignore` 语法的多层黑名单机制（全局黑名单 + 项目级黑名单 + 项目 `.gitignore`）；
- 废弃旧版 `query` 子命令，以新版 `resolve` 子命令替代。

#### 11.1.3 影响范围


| 模块名称     | 原有实现                                     | 变更影响                                                                   |
| -------- | ---------------------------------------- | ---------------------------------------------------------------------- |
| 存储模块     | 仅生成 Markdown 结果文件                        | 新增集中式索引文件 `analysis-index.json` 的生成与维护                                 |
| CLI 命令模块 | 包含 `analyze`、`query`、`config` 三个子命令      | 废弃 `query`，新增 `resolve`，`analyze` 新增 `--skills-providers` 参数           |
| Skill 模块 | 无                                        | 新增 Agent Skill 自动生成与部署能力                                               |
| 文件过滤模块   | 硬编码 `IGNORED_PATTERNS` + 读取 `.gitignore` | 重构为三层黑名单合并机制                                                           |
| 配置模块     | `analyze.exclude_patterns` 控制过滤          | 新增 `skills` 配置段和 `analyze.blacklist` 配置段，废弃 `analyze.exclude_patterns` |
| 常量定义     | `IGNORED_PATTERNS` 常量                    | 废弃该常量，由配置驱动                                                            |


### 11.2 集中式索引文件（analysis-index.json）

#### 11.2.1 索引文件位置与命名

- 文件名：`analysis-index.json`
- 存储位置：解析结果输出目录的根目录下，即 `{输出目录}/analysis-index.json`
  - 默认路径示例：`{项目根目录}/.code-analyze-result/analysis-index.json`
- 索引文件与 `.analysis_metadata.json` 同级存放

#### 11.2.2 索引文件结构

```json
{
  "version": "1.0",
  "projectRoot": "C:/Software/my-project",
  "storageRoot": "C:/Software/my-project/.code-analyze-result",
  "generatedAt": "2026-03-12T10:00:00.000Z",
  "entries": {
    "C:/Software/my-project/src/services/auth.service.ts": {
      "resultPath": "C:/Software/my-project/.code-analyze-result/src/services/auth.service.md",
      "type": "file"
    },
    "C:/Software/my-project/src/services": {
      "resultPath": "C:/Software/my-project/.code-analyze-result/src/services/index.md",
      "type": "directory"
    },
    "C:/Software/my-project/src": {
      "resultPath": "C:/Software/my-project/.code-analyze-result/src/index.md",
      "type": "directory"
    }
  }
}
```

字段说明：


| 字段                         | 类型     | 说明                            |
| -------------------------- | ------ | ----------------------------- |
| `version`                  | string | 索引文件格式版本号                     |
| `projectRoot`              | string | 被解析项目的根目录绝对路径                 |
| `storageRoot`              | string | 解析结果输出目录的绝对路径                 |
| `generatedAt`              | string | 索引文件生成时间（ISO 8601）            |
| `entries`                  | object | 映射表，key 为源码绝对路径，value 为结果信息   |
| `entries[path].resultPath` | string | 对应分析结果 Markdown 文件的绝对路径       |
| `entries[path].type`       | string | 对象类型：`"file"` 或 `"directory"` |


#### 11.2.3 所有路径均为绝对路径

- `projectRoot`、`storageRoot`、`entries` 中的 key 和 `resultPath` 均存储为**绝对路径**，消除任何相对路径歧义；
- 路径分隔符统一使用正斜杠（`/`），即使在 Windows 系统上也使用 `/`，确保跨平台一致性；
- 路径大小写敏感（保持操作系统原始大小写）；
- 路径末尾不含尾部斜杠（目录路径不以 `/` 结尾）。

#### 11.2.4 索引文件生成时机

- 索引文件在**每次解析流程结束后**生成或更新；
- 全量解析：从零构建完整索引，包含所有已生成的文件和目录 Markdown 的映射；
- 增量解析：在现有索引基础上，更新变更文件/目录的映射，删除已不存在的文件/目录的映射，新增新文件/目录的映射；
- 索引文件的生成在所有 Markdown 结果写入完成之后执行，确保索引内容与磁盘文件一致。

#### 11.2.5 索引文件与分片解析的关系

- 索引文件只记录**最终结果**的 Markdown 文件路径；
- 大文件分片解析过程中的中间临时结果不进入索引；
- 大目录分组解析过程中的中间临时结果不进入索引；
- 只有经过完整解析流程（分片→合并→正式写入）后生成的最终 Markdown 文件才被记录到索引中。

#### 11.2.6 索引条目覆盖规则

- 一个源码路径在索引中有且仅有一条记录；
- 第二次解析覆盖第一次的 `resultPath`（实际上路径映射规则固定，通常不会变化，但以最新解析结果为准）；
- 若某个文件在增量解析中被检测为已删除，则从索引中移除对应条目。

### 11.3 解析结果路径查询功能

#### 11.3.1 功能概述

提供两种形式的路径查询能力，共享同一套核心查询逻辑：

1. **CLI `resolve` 子命令**：供人类在终端直接使用；
2. **独立查询脚本**（`scripts/resolve.js`）：内嵌在 Agent Skill 中，供 LLM Agent 调用，不依赖全局安装 `code-analyze`。

#### 11.3.2 输入输出规范

- **输入**：一个绝对路径，指向被解析项目中的某个文件或目录
- **输出**：
  - 若该路径在索引中存在对应记录：输出对应分析结果 Markdown 文件的**绝对路径**（单行纯文本）
  - 若不存在：输出 `N/A`（单行纯文本）
- **退出码**：
  - `0`：查询成功（无论结果是路径还是 `N/A`）
  - `1`：查询失败（索引文件不存在、路径格式非法等异常情况）

#### 11.3.3 路径兼容性处理

由于工具需在 Windows/macOS/Linux 多平台使用，查询逻辑须进行以下路径规范化处理：

- 将输入路径中的反斜杠（`\`）统一转换为正斜杠（`/`）；
- 移除路径末尾的尾部斜杠；
- 路径比较时大小写敏感（不做大小写归一化）；
- 规范化处理后与索引中的 key 进行精确匹配。

#### 11.3.4 CLI `resolve` 子命令

- 命令格式：`code-analyze resolve <absolute-path>`
- 参数：


| 参数名               | 类型     | 必填  | 说明              |
| ----------------- | ------ | --- | --------------- |
| `<absolute-path>` | string | 是   | 需要查询的文件/目录的绝对路径 |


- 选项：


| 选项                     | 说明     | 默认值                       |
| ---------------------- | ------ | ------------------------- |
| `-p, --project <path>` | 项目根路径  | `process.cwd()`           |
| `--output-dir <path>`  | 结果输出目录 | 配置文件中 `global.output_dir` |


- 命令逻辑：
  1. 根据 `--project` 和 `--output-dir` 确定索引文件路径：`{输出目录}/analysis-index.json`
  2. 读取并解析索引文件
  3. 对输入路径进行规范化处理
  4. 在索引 `entries` 中查找匹配条目
  5. 若找到：向 stdout 输出 `resultPath` 绝对路径
  6. 若未找到：向 stdout 输出 `N/A`
  7. 若索引文件不存在或读取失败：向 stderr 输出错误信息，退出码 1
- 示例：

```bash
# 查询文件
code-analyze resolve C:/Software/my-project/src/services/auth.service.ts
# 输出: C:/Software/my-project/.code-analyze-result/src/services/auth.service.md

# 查询目录
code-analyze resolve C:/Software/my-project/src/services
# 输出: C:/Software/my-project/.code-analyze-result/src/services/index.md

# 不存在的路径
code-analyze resolve C:/Software/my-project/src/nonexistent.ts
# 输出: N/A
```

#### 11.3.5 独立查询脚本（scripts/resolve.js）

- 脚本语言：Node.js（与项目技术栈一致，跨平台兼容）
- 调用方式：`node scripts/resolve.js <absolute-path>`
- 脚本为**自包含**的，不依赖全局安装 `code-analyze` CLI，不依赖 `node_modules`
- 脚本读取同目录下的 `resolve-config.json`（见 11.4.5）获取索引文件路径
- 输入输出规范与 11.3.2 完全一致
- 脚本内部仅使用 Node.js 内置模块（`fs`、`path`），不引入任何第三方依赖

#### 11.3.6 查询逻辑伪代码

```
function resolve(inputPath):
    normalizedPath = normalizePath(inputPath)    // 统一分隔符、去尾部斜杠
    indexData = readJSON(indexFilePath)
    if indexData.entries[normalizedPath] exists:
        return indexData.entries[normalizedPath].resultPath
    else:
        return "N/A"
```

#### 11.3.7 旧版 `query` 子命令废弃

- 完全移除 `query` 子命令及其所有相关代码；
- 原 `query` 的功能（查看分析内容）由以下方式替代：
  - 使用 `resolve` 获取 Markdown 文件路径后，直接打开/读取该 Markdown 文件；
  - LLM Agent 通过 Skill 自动完成上述流程。

### 11.4 Agent Skill 自动生成

#### 11.4.1 概述

本工具在执行解析（`analyze`）时，自动生成符合 [Agent Skills 开放标准](https://agentskills.io/specification) 的 Skill，部署到被解析项目的指定目录中。部署完成后，支持 Agent Skills 标准的 AI 编程工具（如 OpenCode、Cursor、Claude Code、Codex 等）可自动发现并使用该 Skill，从而在编码过程中快速查询任意文件或目录的代码解析结果，**而无需重新解析代码**。

#### 11.4.2 Skill 定义


| 属性       | 值                                                                  |
| -------- | ------------------------------------------------------------------ |
| Skill 名称 | `code-query`                                                       |
| Skill 描述 | 查询项目中任意文件或目录的代码解析结果。当需要了解某个源码文件或目录的功能、类定义、接口、全局函数等结构化信息时使用此 Skill。 |
| 兼容性      | 需要 Node.js 运行环境                                                    |


#### 11.4.3 Skill 目录结构

```
code-query/
├── SKILL.md                # 必需：Skill 定义与使用说明
├── resolve-config.json     # 自动生成：索引文件路径等配置
└── scripts/
    └── resolve.js          # 独立查询脚本
```

#### 11.4.4 SKILL.md 内容规范

SKILL.md 文件须包含以下内容：

1. **YAML Frontmatter**（必需）：
  - `name: code-query`
  - `description`：Skill 描述文本（见 11.4.2）
  - `compatibility: Requires Node.js runtime`
2. **Markdown 正文**（必需），至少包含以下章节：
  - **使用场景**：说明何时应使用此 Skill
  - **使用方法**：给出脚本调用命令和参数格式
  - **使用流程**：
  1. 运行脚本获取 Markdown 文件路径
  2. 如果返回值不是 `N/A`，读取该 Markdown 文件内容
  3. Markdown 中包含目标文件/目录的完整结构化解析信息
    输入输出说明**：
    输入：文件或目录的绝对路径
    成功输出：Markdown 结果文件的绝对路径
    不存在时输出：`N/A`
    示例**：至少包含查询文件、查询目录、不存在路径三个示例

#### 11.4.5 resolve-config.json

在 Skill 生成时自动写入，记录查询脚本运行所需的配置信息：

```json
{
  "indexFilePath": "C:/Software/my-project/.code-analyze-result/analysis-index.json"
}
```


| 字段              | 类型     | 说明                          |
| --------------- | ------ | --------------------------- |
| `indexFilePath` | string | `analysis-index.json` 的绝对路径 |


#### 11.4.6 多 AI 工具部署策略

不同 AI 编程工具在项目中搜索 Skill 的目录不同。经调研，各工具支持的项目级 Skill 目录如下：


| AI 工具       | 支持的项目级 Skill 目录                                                        |
| ----------- | ---------------------------------------------------------------------- |
| OpenCode    | `.opencode/skills/`、`.agents/skills/`、`.claude/skills/`                |
| Cursor      | `.cursor/skills/`、`.agents/skills/`、`.claude/skills/`、`.codex/skills/` |
| Claude Code | `.claude/skills/`                                                      |
| Codex       | `.agents/skills/`、`.codex/skills/`                                     |


其中 `.agents/skills/` 是 Agent Skills 标准推荐的通用目录，被 OpenCode、Cursor、Codex 同时支持。Claude Code 是唯一仅支持 `.claude/skills/` 的工具。

**部署规则**：

- 统一使用 `.agents/skills/` 作为通用部署目录，覆盖 OpenCode、Cursor、Codex；
- 对 Claude Code 额外部署一份到 `.claude/skills/`；
- 若多个 provider 映射到同一个目录，只写入一次（去重）；
- Skill 内容（`SKILL.md`、`scripts/resolve.js`、`resolve-config.json`）在所有目录中完全相同。

#### 11.4.7 工具与目录映射表


| Provider 标识 | 部署目录                                 |
| ----------- | ------------------------------------ |
| `opencode`  | `{项目根目录}/.agents/skills/code-query/` |
| `cursor`    | `{项目根目录}/.agents/skills/code-query/` |
| `codex`     | `{项目根目录}/.agents/skills/code-query/` |
| `claude`    | `{项目根目录}/.claude/skills/code-query/` |


- `opencode`、`cursor`、`codex` 三者共享 `.agents/skills/` 目录，部署时只需写入一次；
- 当 providers 列表中包含 `claude` 时，额外写入 `.claude/skills/` 目录。

#### 11.4.8 CLI 参数：`--skills-providers`

在 `analyze` 子命令中新增 `--skills-providers` 选项：


| 选项                          | 说明              | 默认值                              |
| --------------------------- | --------------- | -------------------------------- |
| `--skills-providers <list>` | 逗号分隔的 AI 工具标识列表 | 配置文件中 `skills.default_providers` |
| `--no-skills`               | 跳过 Skill 生成     | `false`                          |


- 示例：

```bash
# 为 OpenCode 和 Claude Code 生成 Skill
code-analyze analyze --path /my-project --skills-providers opencode,claude

# 使用配置文件默认 providers
code-analyze analyze --path /my-project

# 跳过 Skill 生成
code-analyze analyze --path /my-project --no-skills
```

- `--skills-providers` 的值为逗号分隔的 provider 标识，可选值：`opencode`、`cursor`、`claude`、`codex`（不区分大小写）；
- 若传入不可识别的 provider 标识，输出警告信息但不中断解析流程。

#### 11.4.9 配置文件中的默认 providers

在配置文件中新增 `skills` 配置段：

```yaml
skills:
  default_providers:
    - opencode
    - cursor
    - claude
    - codex
```

- `default_providers`：默认为哪些 AI 工具生成 Skill，当 CLI 未传入 `--skills-providers` 时使用此值；
- 默认值包含 `opencode`、`cursor`、`claude`、`codex` 四个主流工具。

#### 11.4.10 Skill 生成时机

- Skill 生成在 `analyze` 命令的解析流程**全部完成**且**索引文件已生成**之后执行；
- 执行顺序：分析 → 写入 Markdown 结果 → 生成 `analysis-index.json` → 生成 Skill 文件；
- Skill 生成失败不影响解析结果的完整性，仅输出警告信息。

#### 11.4.11 Skill 更新策略

- 每次执行 `analyze` 时，若 Skill 文件已存在，则**直接覆盖**（`resolve-config.json` 中的 `indexFilePath` 可能因 `--output-dir` 不同而变化）；
- Skill 中的 `SKILL.md` 和 `scripts/resolve.js` 为固定内容，仅 `resolve-config.json` 会随配置变化而更新。

### 11.5 黑名单过滤规则重构

#### 11.5.1 旧版过滤规则废弃

- 完全废弃 `src/common/constants.ts` 中的 `IGNORED_PATTERNS` 常量；
- 完全废弃配置文件中的 `analyze.exclude_patterns` 配置项；
- 上述旧版过滤逻辑由本节定义的新版黑名单机制完全替代。

#### 11.5.2 新版黑名单机制概述

新版过滤体系由以下三个来源组成，三者合并生效：


| 来源              | 位置                                | 作用域  | 说明           |
| --------------- | --------------------------------- | ---- | ------------ |
| 全局黑名单           | 配置文件 `analyze.blacklist`          | 所有项目 | 用户维护的全局过滤规则  |
| 项目级黑名单          | 被解析项目根目录下的 `.code-analyze-ignore` | 当前项目 | 项目特有的过滤规则    |
| 项目 `.gitignore` | 被解析项目根目录下的 `.gitignore`           | 当前项目 | Git 忽略规则自动纳入 |


**过滤判定流程**：对于每个待处理的文件/目录，依次检查以上三个来源，任一命中即视为应被过滤。

#### 11.5.3 黑名单语法规范

全局黑名单和项目级黑名单（`.code-analyze-ignore`）的语法规则**完全遵循 `.gitignore` 规范**，包括但不限于：

- 每行一个模式（pattern）
- 以 `#` 开头的行为注释
- 以 `!` 开头的行为否定模式（取消匹配）
- 支持 `*`（任意字符）、`*`*（任意层级目录）、`?`（单个字符）通配符
- 以 `/` 结尾表示仅匹配目录
- 以 `/` 开头表示从根目录开始匹配（相对于项目根目录）
- 空行被忽略

示例：

```
# 配置文件
*.yml
*.yaml
*.ini
*.cfg
*.json
*.toml

# 文档文件
*.md
*.txt
*.rst
README*
LICENSE*
CHANGELOG*

# 锁文件和包管理
*.lock
package-lock.json
yarn.lock

# 环境与密钥文件
*.env*
credentials.*
*.pem
*.key

# 文档目录
docs/

# 构建输出
dist/
build/
coverage/

# 依赖目录
node_modules/
.git/

# 解析结果自身
.code-analyze-result/
```

#### 11.5.4 全局黑名单配置

- 配置位置：配置文件（默认 `~/.config/code-analyze/config.yaml`）中的 `analyze.blacklist` 字段
- 类型：字符串数组，每个元素为一条 `.gitignore` 语法的模式
- 作用域：对所有被解析项目生效
- 默认值：内置一组合理的默认黑名单规则（见 11.5.3 示例）

配置文件示例：

```yaml
analyze:
  blacklist:
    - "*.yml"
    - "*.yaml"
    - "*.ini"
    - "*.cfg"
    - "*.json"
    - "*.toml"
    - "*.md"
    - "*.txt"
    - "*.rst"
    - "README*"
    - "LICENSE*"
    - "CHANGELOG*"
    - "*.lock"
    - "package-lock.json"
    - "yarn.lock"
    - "*.env*"
    - "credentials.*"
    - "*.pem"
    - "*.key"
    - "docs/"
    - "dist/"
    - "build/"
    - "coverage/"
    - "node_modules/"
    - ".git/"
    - ".code-analyze-result/"
```

#### 11.5.5 项目级黑名单文件

- 文件名：`.code-analyze-ignore`
- 位置：被解析项目的根目录下
- 语法：完全遵循 `.gitignore` 规范（见 11.5.3）
- 作用域：仅对当前被解析项目生效
- 可选：该文件不是必需的，不存在时仅使用全局黑名单和 `.gitignore`

示例（`.code-analyze-ignore`）：

```
# 项目特有的非代码文件
*.config.js
*.config.ts
jest.config.*
webpack.config.*
rollup.config.*

# 测试辅助文件
__mocks__/
__fixtures__/
*.fixture.*

# 自动生成的代码
*.generated.ts
*.g.dart
```

#### 11.5.6 多层黑名单合并规则

三个来源的合并方式：

1. 先加载全局黑名单（配置文件 `analyze.blacklist`）
2. 再加载项目级黑名单（`.code-analyze-ignore`，若存在）
3. 再加载项目 `.gitignore`（若存在）
4. 三者的模式**顺序拼接**，共同作用于文件过滤判定

合并语义：

- 三个来源的模式按顺序合并为一个统一的规则集；
- 后出现的规则可以通过否定模式（`!` 前缀）覆盖前面的规则（遵循 `.gitignore` 的否定语义）；
- 例如：全局黑名单屏蔽了 `*.json`，但项目级 `.code-analyze-ignore` 中写了 `!tsconfig.json`，则 `tsconfig.json` 不会被过滤。

#### 11.5.7 黑名单与 .gitignore 的关系

- 项目 `.gitignore` 作为三层过滤的第三个来源，自动纳入过滤规则；
- 程序读取并解析项目根目录下的 `.gitignore` 文件（若存在）；
- `.gitignore` 的模式与全局黑名单、项目级黑名单统一合并生效；
- 这意味着 Git 忽略的文件/目录也不会被解析（例如 `node_modules/`、`.git/` 等通常已在 `.gitignore` 中的条目）。

#### 11.5.8 黑名单与二进制文件判定的关系

- 黑名单过滤和二进制文件判定是**两个独立的过滤阶段**，顺序如下：
  1. 先进行黑名单过滤（匹配三层合并后的规则）
  2. 再对通过黑名单的文件进行二进制判定（10.4.1 节定义的逻辑）
- 被黑名单过滤的文件不再进行二进制判定（无需读取文件内容，提升性能）

#### 11.5.9 被黑名单过滤文件的处理

- 与 V2.2 需求 10.4.2 保持一致：
  - 不进行 LLM 解析
  - 不生成对应的 Markdown 结果文件
  - 不计入目录 Markdown 的文件描述列表
  - 不记入索引文件 `analysis-index.json`
  - 以 `filtered` 状态记入进度统计
