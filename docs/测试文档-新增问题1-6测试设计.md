### 12.1 问题 1：非 Git 项目增量解析（第二次不识别文件变更）

#### 12.1.1 测试目标与范围

- **行为目标**：
  - 在非 Git 目录下，`mode=auto`、`mode=incremental` 不能出现“0 工作量伪增量”：文件已修改但第二次运行直接返回旧结果。
  - 在 Git 目录下，`mode=auto` 仍然能正确选择增量模式并只处理变更文件。
- **范围**：
  - 覆盖非 Git / Git 两类项目；
  - 覆盖 `mode=full/auto/incremental` 的组合；
  - 覆盖第一次 / 第二次 / 多次运行的状态演化。

#### 12.1.2 测试分层策略

- **单元测试（unit）**
  - 关注“决策逻辑”：在 `gitCommits=[]`、`mode=auto` 且已有元数据时，最终是否选择增量，以及如何处理 `changedFiles.length === 0`。
  - 关注 `diffCommits` / `getChangedFiles` 在非 Git 场景下是否返回“非 Git 信号”，而不是简单的空数组。
- **集成测试（integration）**
  - 从 `AnalysisAppService.runAnalysis` 入口验证：
    - 非 Git + `mode=auto`：第二次运行会重新处理修改文件（全量或 hash 增量），而不是直接 0 工作量返回；
    - Git + `mode=auto`：能正确感知 Git 变更并执行增量。
- **CLI e2e**
  - 在真实 CLI (`ca analyze`) 下，非 Git 目录多轮运行：
    - stdout 中不应出现“没有检测到变更文件，直接返回现有解析结果”这类误导性文案；
    - 能看见“已处理: x/y 对象”中 x>0 的实际工作量；
    - 结果目录中对应文件的 md/json 内容在第二次运行后发生变化。

#### 12.1.3 关键测试用例概览（含伪代码骨架）

#### 12.1.4 注意事项与扩展方向

- 未来若实现“基于 hash 的非 Git 增量”，应在上述 IT / E2E 用例基础上：
  - 把“第二次必须重新解析”放宽为“第二次必须检查 hash 且在 hash 变化时重新解析”；  
  - 增加对 `changedFiles` 列表中 hash 对应路径的断言。
- 建议对 **多次运行（>3 次）** 的场景单独增加 E2E 用例，验证元数据/缓存不会积累出新的伪增量状态。

### 12.2 问题 2：并发分析与深层目录结构（父目录长期占用线程感知）

#### 12.2.1 测试目标与范围

- **行为目标**：
  - 量化当前实现下“有效并发度”（任意时刻活跃对象数量），识别父目录长期处于 active 状态的模式；
  - 确保在深层目录 + 高并发配置下，不会出现“长期仅 1～2 个活跃对象”的严重并发退化；
  - 验证 CLI `[当前对象]` 列表与内部 active 集合的一致性。
- **范围**：
  - 深层、多分支目录树（5～10 层）；
  - 不同 `concurrency` 配置（1, 2, 4, 8 等）；
  - 全量分析场景（目前 WorkerPool 尚未接入）。

#### 12.2.2 测试分层策略

- **单元测试（unit）**
  - 基于 `AnalysisService.fullAnalysis` + `onObjectStarted/onObjectCompleted` 回调：
    - 维护 `activeSet` 和对象生命周期（start/end 时间），统计 `maxConcurrency` / `avgConcurrency`；
    - 度量父目录与子文件的活跃时长差异。
- **集成测试（integration）**
  - 在测试侧模拟 App 层的 active 集合与“当前对象列表”裁剪逻辑：
    - 验证 `displayLines`（当前对象列表）是 `activeSet` 的有序子集；
    - 在深层树中统计父目录路径在 `displayLines` 中出现的次数。
- **CLI e2e**
  - 解析真实 stdout 中的“当前对象:” 块：
    - 统计每个 snapshot 中的行数（并发度感知）；
    - 统计特定父目录路径出现次数与持续时间。

#### 12.2.3 关键测试用例概览（含伪代码骨架）

```ts
  writeDummyTree(root, {

    'b/b1/file4.ts': '4',
    'c/c1/c1_1/c1_1_1/file5.ts': '5',

  const activeSet = new Set<string>();
  const startAt: Record<string, number> = {};
  const endAt: Record<string, number> = {};

  await svc.fullAnalysis({
    projectRoot: root,
    depth: -1,
    concurrency: 8,

      activeSet.delete(k);

    onProgress: () => {},

  const rootKey = 'directory:.';
  const rootLifetime = endAt[rootKey] - startAt[rootKey];
  // 与平均文件时长对比
});
```

### 12.3 问题 3：CLI 进度条 / 当前对象 / Tokens 输出不符合预期

#### 12.3.1 测试目标与范围

- **目标**：
  - 进度条、当前对象列表、Tokens 统计在不同运行场景（TTY/非 TTY、full/incremental、小/大项目、异常中断）下输出格式稳定、含义明确；
  - 不再出现“多块进度区域堆叠”“当前对象块重复刷屏”“Tokens 行难以解析”等问题。
- **范围**：
  - CLI 层输出（stdout/stderr）；
  - UI 封装组件（`ProgressBar`、UI adapter）；
  - App 层 `handleObjectCompleted` 与 `onProgress` 回调。

#### 12.3.2 测试分层策略

- **单元（UI 组件层）**
  - 针对 `ProgressBar` 在 TTY/非 TTY 下的写入模式（`\r` 覆盖 vs 纯文本换行）做精确断言；
  - 验证“当前对象”“Tokens”展示通过统一 UI adapter，避免直接散落 `console.log`。
- **集成（App + UI）**
  - 使用假的 UI adapter 捕获 `handleObjectCompleted` / `onProgress` 的调用序列：
    - 验证当前对象列表与内部 active 集合一致；
    - 验证 Tokens 更新频率与数值的单调性。
- **CLI e2e**
  - 捕获 stdout，按块解析：
    - 统计进度条行数、当前对象块数、Tokens 行数；
    - 验证它们的数量与项目规模、并发配置成合理关系，而不是指数级增长。

### 12.4 问题 4：初始进度条 total=100（魔法数字）

#### 12.4.1 测试目标与范围

- **目标**：
  - 禁止任何层面（UI/CLI）使用硬编码 `100` 作为初始 total；
  - 确保首帧进度条 total 来自真实对象总数或明确的“0/未初始化”语义；
  - 在 CLI e2e 中，首条“解析进度”行的 total 与扫描到的对象总数一致。
- **范围**：
  - `src/cli.ts` 启动逻辑；
  - `onTotalKnown` 回调前后进度条行为；
  - CLI e2e 首帧日志。

#### 12.4.2 测试分层策略

- **单元（UI/CLI 封装）**
  - 针对 `ProgressBar.start` 的调用方（CLI 层）增加测试，禁止 `total=100` 之类固定值在“真实 total 未知”时被使用。
- **集成（App + UI）**
  - 模拟 `onTotalKnown(total)` 调用前后的输出序列：
    - 在 `onTotalKnown` 之前不输出任何包含 `/100` 的进度条；
    - `onTotalKnown` 被调用后，才输出 `/真实 total`。
- **CLI e2e**
  - 抓取 stdout 中首条 progress 行，解析 `已处理: 0/total`，断言 `total != 100`，且为真实值（或 0）。

#### 12.4.3 关键测试用例概览（含伪代码骨架）

### 12.5 问题 5：结果目录中多出 per-file/per-dir JSON 文件

#### 12.5.1 测试目标与范围

- **目标**：
  - 建立对结果目录结构的“白名单契约”：只允许特定类型文件（如 `.md`、集中 `analysis-index.json`、`.analysis_metadata.json` 等）；
  - 防止内部中间产物（per-file/per-dir json、缓存文件）泄露到用户可见结果目录；
  - 为未来迁移缓存策略（单独 cacheDir）提供稳定回归基线。
- **范围**：
  - `LocalStorageService` 的落盘行为；
  - `AnalysisService/AnalysisAppService` 在 runAnalysis 后的目录结构；
  - CLI `--output-dir` 自定义输出路径下的结果树。

#### 12.5.2 测试分层策略

- **单元（LocalStorageService）**
  - 明确 `saveFileAnalysis/saveDirectoryAnalysis` 写入的文件集合；
  - 测试 `getFileAnalysis/getDirectoryAnalysis` 在 json 缺失时的容错行为。
- **集成（AnalysisService/AnalysisAppService）**
  - 在一次 full 分析后，对 `.code-analyze-result` 递归遍历，使用扩展名白名单过滤；
  - 若允许 per-file json，需约束 json 文件位置与命名规则；若不允许，则直接 fail。
- **CLI e2e**
  - 从 CLI 调用后，对结果目录做快照，并存储一份 golden snapshot（可选）；
  - 对比快照 / 白名单校验，保证后续更改不会引入额外类型文件。

### 12.6 问题 6：大项目解析时 CLI crash（Node 堆 OOM）

#### 12.6.1 测试目标与范围

- **目标**：
  - 在接近真实大项目（数百～上千对象）场景下，系统不会以 `JavaScript heap out of memory` 形式崩溃；
  - 未来引入 token / 内存保护策略后，超限时以业务错误（带错误码和友好信息）优雅失败；
  - CLI 层 exitCode 与 stderr 输出遵循统一规范。
- **范围**：
  - LLM usage 统计与上限控制；
  - `AnalysisAppService.runAnalysis` 在高负载下的行为；
  - CLI 在大项目 + mock LLM 环境下的端到端表现。

#### 12.6.2 测试分层策略

- **单元**
  - `LLMUsageTracker`：大量调用场景下正确统计 tokens，未来添加上限时在超过 maxTokens 处抛出 `ResourceLimitError`。
  - `LlmService` / `AnalysisService`：在 `ResourceLimitError` 发生时不 crash，而是向上传递业务错误。
- **集成**
  - 构造 200～600 对象的“精简大项目”，使用 Mock LLM（高 usage 数、体积可控），验证：
    - 正常路径：`runAnalysis` 可顺利完成；
    - 限制路径：低 maxTokens 下，`runAnalysis` 返回资源限制造成的业务错误。
- **CLI e2e**
  - 通过 spawn CLI，对大项目执行 `analyze`：
    - 正常：exitCode=0、stderr 不含 OOM 文案；
    - 限制：exitCode=特定错误码（如 101）、stderr 中包含 `RESOURCE_LIMIT`，且不含 OOM。

## 13. 测试策略与覆盖分析总结（面向问题 1～6）

本小节从测试体系视角，对问题 1～6 的共性盲区和建设优先级进行归纳，帮助后续在补充具体用例之外，形成可持续演进的测试策略。

### 13.1 从测试视角重述问题共性盲区

- **大项目场景 & 复杂结构缺乏系统覆盖**  
  多个问题只在真实大仓库（多模块、多语言、深层目录）下暴露，说明当前测试几乎只覆盖“小而干净”的项目。需要引入代表性大项目 fixture 和规模梯度（小/中/大）测试。
- **非 Git / 异常 Git 状态覆盖不足**  
  现有测试多依赖“干净 Git 仓”，对非 Git、子目录执行、detached head、工作区脏等真实场景缺少系统性组合，导致增量/索引逻辑在这些场景下极易出错。
- **CLI UI 行为未被视为契约**  
  进度条、当前对象、Tokens 输出长期缺乏独立测试，任何改动都可能破坏用户可观测性。需要把 CLI UI 抽象为可测试接口并建立稳定断言。
- **结果目录结构未被契约化**  
  落盘策略、目录层级、清理策略长期被视作实现细节，缺少“白名单约束 + 多轮运行对比”的测试，导致历史产物、cache 文件容易泄露到结果目录。
- **资源极限与失败模式建模不足**  
  对 LLM 超时、token 超配、OOM 等资源问题缺乏统一错误模型和系统性测试，各模块对异常处理风格不统一，容易出现静默失败或崩溃。
- **配置矩阵与默认值缺乏组合测试**  
  CLI 参数、配置文件、环境变量的优先级与默认值未形成“配置矩阵”式测试，边界配置（部分缺失/冲突）行为不清晰。

### 13.2 测试建设优先级（摘要）

1. **结果目录结构与落盘策略（P0）**  
   - 以白名单 + 快照方式锁定 `.code-analyze-result` 结构，优先补齐单元/集成/CLI e2e 用例。
2. **非 Git / Git + 增量/全量组合（P0）**  
   - 系统化构造 Git 状态矩阵，防止增量逻辑在真实工作流下失效。
3. **CLI 进度条 / 当前对象 / Tokens 行为（P1）**  
   - 建立 UI 契约，覆盖 TTY/非 TTY、成功/失败等场景。
4. **大项目 + 高 token 调用资源鲁棒性（P1）**  
   - 使用 Mock LLM + 大项目 fixture，避免 OOM，验证资源保护行为。
5. **配置矩阵与默认值一致性（P2）**  
   - 定义配置优先级矩阵，用测试锁定不同入口的一致行为。
